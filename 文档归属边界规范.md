# 文档归属边界规范

> 目标：判断一份文档应该放在 Obsidian 知识库、具体项目仓库、VPS 目录、项目管理工具，还是 1Password。

## 1. 总原则

Obsidian 保存长期沉淀，项目仓库保存交付事实，VPS 目录保存服务器独特信息，项目管理工具保存执行过程，1Password 保存秘密。

判断一份文档归属时，优先问四个问题：

1. 换一个项目以后还会看吗？
2. 是否必须和某个代码版本严格对应？
3. 它是在解释“为什么这样做”，还是说明“当前项目怎么运行”？
4. 它是否包含密码、token、私钥、生产密钥等秘密？

## 2. Obsidian 适合保存什么

Obsidian 适合保存可复用、可沉淀、跨项目仍有价值的文档。

适合放入 Obsidian：

- 前期技术论证
- 技术选型比较
- 架构方案推演
- 项目启动前的方案草稿
- 决策记录，尤其是为什么选 A 不选 B
- 通用开发规范
- 通用运维规范
- 跨项目故障复盘
- 个人学习笔记
- 工具使用方法
- 可复用模板
- 对外或内部可长期复用的方案文档

推荐目录：

```text
笔记/
  文档归属边界规范.md
  Git同步规范.md
  SSH密钥管理.md

项目/
  数据存储与PaaS部署/
    技术论证.md
    选型比较.md
    架构方案.md
    决策记录.md
    复盘.md
```

## 3. 项目仓库适合保存什么

项目仓库适合保存和当前项目交付、代码版本、运行方式强绑定的文档。

适合放入项目仓库：

- `README.md`
- API 文档
- 数据库 schema 说明
- 环境变量说明
- 部署说明
- 本项目模块说明
- 测试说明
- Release notes
- Changelog
- 当前代码版本对应的设计说明
- 当前代码版本对应的迁移说明
- 项目内脚本使用说明
- 与 CI/CD、Docker、Compose、Makefile 等直接配套的说明

推荐目录：

```text
project-repo/
  README.md
  docs/
    api.md
    deploy.md
    database.md
    env.md
    release.md
    module-design.md
```

项目仓库里的文档应回答：

- 这个项目现在怎么安装？
- 这个项目现在怎么运行？
- 当前接口是什么？
- 当前配置项有哪些？
- 当前版本怎么部署、测试、回滚？

## 4. VPS 目录适合保存什么

VPS 目录只保存服务器独特信息，不保存通用知识。

适合放入 VPS 目录：

- 服务器用途
- 操作系统版本
- 云厂商、地域、规格
- 部署了哪些服务
- 端口占用
- 域名绑定
- 证书状态
- 重要目录结构
- systemd 服务名
- 这台机器独有的历史问题
- 这台机器独有的临时处置记录
- 这台机器和其他机器不同的配置差异

不适合放入 VPS 目录：

- Git 使用规范
- SSH 通用操作方法
- Docker 通用部署规范
- Nginx 通用配置原则
- 项目级技术方案
- 可迁移到其他服务器的运维经验

判断标准：

> 只要换一台服务器也适用，就不要放 VPS；只有这台服务器独有，才放 VPS。

推荐目录：

```text
VPS-AL03/
  deploy-inventory.md
  server-profile.md
  incident-log.md
```

服务器档案可以链接到通用文档：

```md
# VPS-AL03

## 基本信息

- 用途：生产环境
- 系统：Ubuntu 22.04
- 主要目录：/opt/web2py_yf

## 相关文档

- [[Git同步规范]]
- [[SSH密钥管理]]
- [[Docker部署规范]]
```

## 5. 项目管理工具适合保存什么

项目管理工具保存执行过程，不承担长期知识沉淀。

适合放入项目管理工具：

- 任务拆分
- 排期
- 进度跟踪
- Issue
- Bug
- 临时 TODO
- 验收状态
- 责任人
- 看板流转

不建议把大量任务进度塞入 Obsidian。Obsidian 可以保存阶段复盘和关键决策，但不负责替代看板。

## 6. 1Password 适合保存什么

1Password 保存秘密，Obsidian 和项目仓库只记录引用位置。

适合放入 1Password：

- VPS 登录密码
- SSH 私钥 passphrase
- API Key
- Token
- 数据库密码
- 后台账号密码
- 2FA 备用码
- 恢复码
- 生产密钥

Obsidian 可以这样写：

```md
## 登录方式

- SSH 用户：deploy
- 登录密钥：见 1Password 条目 `aliyun-prod SSH`
```

不要在 Obsidian 或项目仓库写：

```text
password=xxxxx
token=xxxxx
-----BEGIN OPENSSH PRIVATE KEY-----
```

## 7. 放置决策表

| 文档类型 | 推荐位置 | 原因 |
| --- | --- | --- |
| 前期技术论证 | Obsidian | 可复用，记录思考过程 |
| 技术选型比较 | Obsidian | 跨项目可参考 |
| 架构方案草稿 | Obsidian | 适合推演和链接 |
| 最终架构图，且与代码版本绑定 | 项目仓库 | 应随代码演进 |
| README | 项目仓库 | 当前项目入口 |
| API 文档 | 项目仓库 | 与代码版本绑定 |
| 数据库说明 | 项目仓库 | 与 schema 绑定 |
| 部署说明 | 项目仓库 | 与当前交付方式绑定 |
| 通用部署规范 | Obsidian | 跨项目复用 |
| 某台 VPS 的端口清单 | VPS 目录 | 服务器独特事实 |
| SSH 通用配置方法 | Obsidian | 通用知识 |
| SSH 私钥 passphrase | 1Password | 秘密 |
| 任务排期 | 项目管理工具 | 执行过程 |
| Bug 列表 | 项目管理工具 | 执行过程 |
| 项目复盘 | Obsidian | 长期沉淀 |

## 8. 推荐工作流

### 8.1 项目前期

1. 在 Obsidian 写技术论证、选型比较、方案推演。
2. 形成关键决策后，保留决策记录。
3. 项目立项后，在项目仓库建立 `README.md` 和 `docs/`。

### 8.2 项目落地

1. 与代码版本强绑定的说明进入项目仓库。
2. 任务、Bug、排期进入项目管理工具。
3. 通用经验和可复用规范回流 Obsidian。

### 8.3 服务器现场记录

1. 服务器独有事实写入对应 VPS 目录。
2. 可复用的运维方法整理进 Obsidian 通用笔记。
3. 密码、token、私钥 passphrase 写入 1Password。

### 8.4 项目结束

1. 项目仓库保留交付文档和版本事实。
2. Obsidian 保存复盘、踩坑、架构经验。
3. VPS 目录保留最终部署事实和历史事件。

## 9. 快速判断口诀

```text
为什么这样做 -> Obsidian
这个项目现在怎么跑 -> 项目仓库
这台服务器有什么不同 -> VPS 目录
什么时候谁来做 -> 项目管理工具
真正的秘密 -> 1Password
```
