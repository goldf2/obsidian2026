# 菜单架构统一规范(供参考)

> 版本: v2.0  
> 日期: 2026-06-18  
> 适用范围: cxmj 类 web2py 内部系统的新 app、新页面、存量 app 菜单治理  
> 当前口径: 本文是菜单架构的当前执行文档,供参考

本文定义页面可显示的菜单类型、结构上限、数据来源和渲染边界。核心原则是: **菜单结构固定，展示位置可调整；系统导航和业务导航分离；toolbar 可选且由页面定义。**

## 1. 总原则

- 顶部固定导航是系统级导航，只放全局入口、跨 app 工具和用户入口。
- Mega menu / 网站地图是保底找路工具，也可以作为全局入口总览；它使用固定的二层分组结构。
- 侧面导航是当前 app 或当前业务域的外部业务导航，最多两层。
- 面包屑导航只显示当前位置的垂直路径，不增加菜单层级。
- 页内导航是当前页面内部内容或视图切换，最多两层。
- Toolbar 是页面动作区或页面快捷区，完全可选，由当前页面定义，不计入菜单层级。
- 菜单内容和结构由系统统一维护，展示位置可以由布局或用户偏好决定。
- 权限过滤发生在菜单数据生成阶段，但目标页面仍必须做后端权限校验。

## 2. 页面菜单类型

| 类型 | 推荐变量 | 生成者 | 用途 | 层级上限 | 是否可选 |
| --- | --- | --- | --- | --- | --- |
| 顶部固定导航 | `response.system_nav` / 兼容 `response.menu` | `service_center` | 系统级全局入口 | 二层 | 否 |
| Mega menu / 网站地图 | `response.global_menu` / `response.app_launcher` | `service_center` | 全局找路、跨 app 入口总览 | 二层 | 建议有 |
| 侧面导航 | `response.side_nav` / 兼容 `response.sub_menu` | 当前 app 或统一菜单服务 | 当前 app/业务域同级和次级页面 | 二层 | 可按展示模式隐藏 |
| 面包屑导航 | `response.breadcrumb` | 当前路由、菜单树或页面 | 当前页面垂直路径 | 不作为菜单层级计算 | 建议有 |
| 页内导航 | `response.page_nav` / 兼容 `response.page_menu` | 当前页面或当前功能模块 | 当前页面内部内容、状态、视图切换 | 二层 | 按页面需要 |
| Toolbar | `response.toolbar` | 当前页面 | 当前页面动作或强相关快捷跳转 | 不计入菜单层级 | 可选 |

兼容规则:

```python
response.menu = response.system_nav
response.sub_menu = response.side_nav
response.page_menu = response.page_nav
```

旧变量可以继续渲染，但新规范应优先使用语义更清晰的变量名。

## 3. 顶部固定导航

顶部固定导航回答的是: **系统有哪些全局入口？**

允许放:

- 系统首页。
- 应用 / 菜单入口，用于打开 Mega menu 或网站地图。
- 站内通知。
- 站内信。
- 分公司/组织切换。
- 系统管理入口。
- 用户菜单: 登录、退出、个人资料、修改密码。
- 少量真正跨 app 的全局工具。

禁止放:

- 普通业务模块列表。
- 当前 app 内部页面。
- 新增、编辑、删除、导入、导出、审核、刷新等页面动作。
- 只服务某个页面的快捷跳转。

顶部固定导航内容基本固定，不随当前业务 app 改变。

## 4. Mega Menu / 网站地图

Mega menu 回答的是: **用户不知道去哪里时，系统能去哪里？**

它采用大型二层结构:

```text
分类头，不跳转
  入口项，可点击
  入口项，可点击
```

第一层是分类头，只用于组织信息，不是实际页面；第二层是可点击入口。

适合放:

- 跨 app 入口。
- 跨业务域入口。
- 关键模块首页。
- 网站地图入口。
- 用户常用但不属于当前页面的系统级功能入口。

禁止放:

- 新增、编辑、删除、导入、导出、审核等页面动作。
- 当前 app 的完整深层业务侧栏。
- 三级及以上菜单。
- 业务流程步骤。

推荐数据结构:

```python
response.global_menu = [
    dict(
        key="quality",
        title=T("质量管理"),
        icon="fas fa-check-circle",
        description=T("检验、异常、质量报表"),
        items=[
            dict(
                key="quality.dashboard",
                title=T("质量看板"),
                url=URL("quality", "default", "dashboard"),
                icon="fas fa-chart-line",
                permission="quality_view",
            ),
            dict(
                key="quality.incoming",
                title=T("来料检验"),
                url=URL("quality", "incoming", "index"),
                icon="fas fa-clipboard-check",
                permission="quality_incoming_view",
            ),
        ],
    ),
]
```

显示规则:

- 分类头没有 URL，不跳转。
- 分类下没有任何可见入口时，隐藏整个分类。
- 入口项必须有稳定 `key`、标题、URL 和权限约束。
- 入口项可带图标和短说明。
- 移动端可降级为抽屉或网站地图列表。

## 5. 侧面导航

侧面导航回答的是: **我在当前业务 app 里有哪些同级和次级页面？**

侧面导航最多两层:

```text
业务模块
  业务页面
```

适合放:

- 当前 app 首页。
- 当前 app 的业务模块。
- 当前模块下的主要业务页面。
- 当前 app 的报表、配置、历史记录等页面入口。

禁止放:

- 顶部系统功能，例如通知、站内信、用户退出、分公司切换。
- 页面动作，例如新增、导入、导出、审核。
- 超过二级的业务菜单。
- 与当前 app 无关的跨系统入口。

示例:

```text
产品库存
  产品列表
  低库存预警
缺货督促
  缺货列表
  紧急预警
  超期预警
发货管理
  发货记录
  发货计划
  发车时刻表
  车辆管理
  统计分析
导入导出
```

如果用户选择 Mega menu 为主要展示方式，侧面导航可以隐藏、折叠，或只显示当前模块分支；但菜单结构本身不变。

## 6. 面包屑导航

面包屑回答的是: **我现在在哪里？**

面包屑显示当前页面的垂直路径，例如:

```text
质量管理 > 检验管理 > 来料检验 > 检验单详情
```

规则:

- 面包屑用于显示路径，不承担主菜单职责。
- 面包屑可以包含可点击的上级路径。
- 详情页、编辑页、审核页可以出现在面包屑中，但不一定出现在菜单中。
- 面包屑不计入菜单层级上限。

## 7. 页内导航

页内导航回答的是: **当前页面内部有哪些内容、状态或视图？**

页内导航最多两层。常规页面优先一层；只有复杂页面才使用两层。

适合放:

- 全部、待处理、已完成、已归档。
- 列表、统计、日志、附件。
- 基本信息、检验项、审批记录、操作日志。
- 用户视图、部门视图、日期视图。

禁止放:

- 当前 app 的主业务模块。
- 跨 app 入口。
- 顶部系统功能。
- 新增、导入、导出、审核等动作。

示例:

```text
发货记录页
  列表 / 看板 / 统计
  全部 / 待发货 / 已发货 / 异常
```

如果还需要更多层级，优先使用筛选器、表格分组、树表、折叠面板、步骤条或详情页 Tab，不继续增加菜单层级。

## 8. Toolbar

Toolbar 回答的是: **当前页面能快速做什么，或能快速跳到哪些强相关功能？**

Toolbar 是页面动作区或页面快捷区，完全可选，由当前页面定义。它可以和顶部栏、页面标题、表格上方同排显示，但语义上不属于菜单层级。

适合放:

- 新增、编辑、删除、保存、提交、审核、退回、作废。
- 导入、导出、下载模板、打印、刷新。
- 批量操作。
- 与当前页面强相关的快捷跳转，例如从产品列表跳到库存预警、从发货记录跳到发货计划。

禁止放:

- 系统首页、网站地图、站内信、通知、退出登录。
- 当前 app 的完整业务菜单。
- 跨业务域入口。
- 页内 Tab 切换。

推荐结构:

```python
response.toolbar = [
    dict(
        key="create",
        label=T("新增"),
        icon="fas fa-plus",
        url=URL("records", "create"),
        style="primary",
        permission="shipping_create",
    ),
    dict(
        key="export",
        label=T("导出"),
        icon="fas fa-download",
        url=URL("records", "export"),
        style="secondary",
        permission="shipping_export",
    ),
]
```

权限规则:

- 按钮隐藏只是体验优化，不是安全措施。
- 每个 toolbar 动作必须有后端权限校验。
- 批量动作必须逐条校验数据权限。
- 对象动作可以单独定义为 `row_actions`，不要把所有单行动作都放入页面 toolbar。

## 9. 展示模式

菜单结构固定，展示位置可以调整。布局或用户偏好可以选择:

| 模式 | 说明 |
| --- | --- |
| `mega` | 顶部系统导航 + Mega menu 作为主要入口 |
| `sidebar` | 顶部系统导航 + 侧面导航作为主要业务导航 |
| `both` | 顶部系统导航 + Mega menu + 当前 app 侧面导航 |
| `compact` | 顶部系统导航 + 折叠侧栏或移动端抽屉 |

无论使用哪种模式，数据结构和权限过滤必须一致。

### 9.1 桌面端同排展示

顶部固定导航、页内导航和 toolbar 可以在桌面端显示在同一条顶部视觉栏或页面标题栏中，但必须保留清晰的逻辑分区。

允许的组合:

```text
左侧: 顶部固定导航
中间: 页内导航或当前页面标题
右侧: Toolbar / 用户菜单
```

要求:

- 顶部固定导航和页内导航之间必须有分隔符、留白、背景色、边框或字体颜色差异。
- 页内导航和 toolbar 之间必须有视觉区隔，不能混成一组按钮。
- 顶部固定导航使用系统级样式，页内导航使用页面级样式。
- Toolbar 可以靠右显示，但不能和用户菜单混淆。
- 同一视觉行承载多个逻辑类型时，DOM 和变量仍必须分开渲染。

推荐视觉表达:

```text
[系统首页] [应用] [通知] | [全部] [待处理] [已完成]              [新增] [导出] [刷新] [用户]
```

其中:

- `系统首页 / 应用 / 通知 / 用户` 来自顶部固定导航。
- `全部 / 待处理 / 已完成` 来自页内导航。
- `新增 / 导出 / 刷新` 来自 toolbar。

### 9.2 移动端收缩

移动端优先保证内容区域可用，菜单和动作入口收缩为明确的浮动或折叠入口。

推荐规则:

- 顶部固定导航收缩到右上角按钮，点击后弹出系统导航、用户菜单和 Mega menu 入口。
- 页内导航收缩到右下角按钮，点击后弹出当前页面内部视图切换。
- Toolbar 如存在，可以并入右下角页内面板，也可以作为页面底部操作条；由页面复杂度决定。
- 侧面导航在移动端收缩为抽屉，或由 Mega menu / 网站地图承担找路功能。

右上角按钮只处理系统级导航；右下角按钮只处理当前页面上下文。两者不得混用。

```text
移动端右上: 系统导航 / 应用 / 通知 / 用户
移动端右下: 页内导航 / 当前页面快捷动作
```

## 10. 层级上限

- 顶部固定导航: 一层。
- Mega menu / 网站地图: 二层，分类头 + 入口项。
- 侧面导航: 二层，同级模块 + 次级页面。
- 页内导航: 二层，当前页面内容组 + 状态/视图。
- 面包屑: 显示路径，不算菜单层级。
- Toolbar: 页面动作或快捷区，不算菜单层级。

超过这些层级时，不继续增加菜单深度，优先使用:

- 页内选项卡。
- 筛选器。
- 树形列表或树表。
- 表格分组。
- 折叠面板。
- 步骤条。
- 搜索。
- 拆分页面或拆分业务模块。

## 11. 存量页面问题样本

`shipping_management/default/index` 代表一种需要治理的旧模式:

- 顶部栏混入了业务模块。
- 顶部下拉混入了新增、创建、导入、导出等动作。
- 缺少回到系统首页或平台首页的入口。
- 缺少 Mega menu / 网站地图，用户无法全局找路。
- 缺少侧面导航，无法稳定进行同级业务跳转。

推荐改造:

- 顶部固定导航只保留系统级入口。
- Mega menu 展示全局二层入口。
- 侧面导航展示 shipping_management 的业务模块和次级页面。
- 页内导航展示当前页面内部视图。
- Toolbar 可选，只放当前页面动作和强相关快捷跳转。

## 12. 存量菜单治理顺序

1. 盘点每个 app 当前使用的顶部导航、Mega menu、侧栏、面包屑、页内导航、toolbar 和数据库菜单。
2. 先统一顶部固定导航来源，清理 app 内重复顶部菜单。
3. 从顶部固定导航中移除普通业务菜单，只保留系统级和跨模块入口。
4. 建立 `response.global_menu`，用于 Mega menu 和网站地图。
5. 把当前 app 的普通业务菜单迁到 `response.side_nav` 或兼容 `response.sub_menu`。
6. 把当前页面内部视图迁到 `response.page_nav` 或兼容 `response.page_menu`。
7. 把新增、编辑、导入、导出、审核等动作迁到 `response.toolbar`。
8. 对 iframe 历史入口标记状态: 保留、改造、归档、删除。
9. 最后清理无入口页面、重复入口和失效 URL。

## 13. 新 app 接入最小清单

- [ ] 已接入 `service_center` 统一认证。
- [ ] 顶部固定导航未混入普通业务菜单。
- [ ] 全局入口已登记到 `response.global_menu`。
- [ ] Mega menu / 网站地图最多两层，第一层分类头不跳转。
- [ ] 当前 app 业务导航写入 `response.side_nav` 或 `response.sub_menu`。
- [ ] 侧面导航最多两层。
- [ ] 当前页面内部导航写入 `response.page_nav` 或 `response.page_menu`。
- [ ] 页内导航最多两层。
- [ ] Toolbar 如存在，由当前页面定义。
- [ ] 桌面端同排展示时，顶部固定导航、页内导航和 toolbar 有分隔符或颜色区分。
- [ ] 移动端顶部固定导航收缩到右上角，页内导航收缩到右下角。
- [ ] 菜单入口和 toolbar 动作都有后端权限校验。
- [ ] 菜单 key 稳定且不重复。

## 14. 文档登记

每个项目应在 `projects/<web2py_service>/菜单治理清单.md` 或项目级导航文档中记录:

- 顶部固定导航入口。
- Mega menu / 网站地图二层结构。
- 当前 app 侧面导航。
- 页面面包屑规则。
- 页内导航。
- Toolbar 使用位置。
- 页面路由和权限。
- 存量菜单清理状态。

如果本文与历史设计文档冲突，以本文为准。
