# 菜单架构统一规范简略版

> 版本: v2.0 简略版  
> 日期: 2026-06-18  
> 适用范围: cxmj 类 web2py 内部系统的新 app、新页面、存量 app 菜单治理  
> 详细说明: 参见 `菜单架构统一规范.md`

本文保留菜单架构的执行口径，删除长示例和重复解释。日常开发、评审和治理优先看本文；遇到争议再查看详细版。

## 1. 核心原则

- 系统导航和业务导航分离。
- 菜单结构统一，展示位置可调整。
- 顶部固定导航只放系统级入口。
- Mega menu / 网站地图只做全局找路。
- 侧面导航只放当前 app 或当前业务域入口。
- 页内导航只放当前页面内部内容、状态或视图。
- 新增、编辑、删除、导入、导出、审核等动作放入 toolbar，不放入菜单。
- 面包屑只显示当前位置，不承担菜单职责。
- Toolbar 可选，由页面定义，不计入菜单层级。
- 菜单隐藏只是体验优化，目标页面和动作仍必须做后端权限校验。

## 2. 菜单类型

| 类型 | 推荐变量 | 生成者 | 用途 | 层级上限 |
| --- | --- | --- | --- | --- |
| 顶部固定导航 | `response.system_nav` / 兼容 `response.menu` | `service_center` | 系统级全局入口 | 一层 |
| Mega menu / 网站地图 | `response.global_menu` / `response.app_launcher` | `service_center` | 全局入口总览 | 二层 |
| 侧面导航 | `response.side_nav` / 兼容 `response.sub_menu` | 当前 app 或统一菜单服务 | 当前 app/业务域导航 | 二层 |
| 面包屑导航 | `response.breadcrumb` | 当前路由、菜单树或页面 | 当前页面路径 | 不计入层级 |
| 页内导航 | `response.page_nav` / 兼容 `response.page_menu` | 当前页面或功能模块 | 当前页面内部视图 | 二层 |
| Toolbar | `response.toolbar` | 当前页面 | 页面动作或强相关快捷跳转 | 不计入层级 |

兼容赋值:

```python
response.menu = response.system_nav
response.sub_menu = response.side_nav
response.page_menu = response.page_nav
```

## 3. 放置规则

### 顶部固定导航

只放系统级入口:

- 系统首页。
- 应用入口或网站地图入口。
- 通知、站内信。
- 分公司/组织切换。
- 系统管理。
- 用户菜单: 登录、退出、个人资料、修改密码。
- 少量真正跨 app 的全局工具。

不得放普通业务模块、当前 app 内部页面、页面动作和页面快捷跳转。

### Mega menu / 网站地图

用于全局找路，结构固定为二层:

```text
分类头，不跳转
  入口项，可点击
```

要求:

- 第一层是分类头，不配置 URL。
- 第二层是入口项，必须有稳定 `key`、标题、URL 和权限约束。
- 分类下没有可见入口时隐藏整个分类。
- 不放页面动作、流程步骤、三级菜单或当前 app 的完整深层侧栏。

### 侧面导航

用于当前 app 或当前业务域，最多二层:

```text
业务模块
  业务页面
```

可放当前 app 首页、业务模块、主要业务页面、报表、配置、历史记录。

不得放顶部系统功能、页面动作、跨系统入口或超过二级的菜单。

### 面包屑

用于显示当前位置，例如:

```text
质量管理 > 检验管理 > 来料检验 > 检验单详情
```

详情页、编辑页、审核页可以出现在面包屑中，但不一定出现在菜单中。

### 页内导航

用于当前页面内部内容、状态或视图切换，优先一层，复杂页面最多二层。

常见内容:

- 全部、待处理、已完成、已归档。
- 列表、看板、统计、日志、附件。
- 基本信息、审批记录、操作日志。

不得放 app 主业务模块、跨 app 入口、系统功能或页面动作。

### Toolbar

用于当前页面动作或强相关快捷跳转，可选。

可放:

- 新增、编辑、删除、保存、提交、审核、退回、作废。
- 导入、导出、下载模板、打印、刷新。
- 批量操作。
- 强相关快捷跳转。

要求:

- 每个动作必须有后端权限校验。
- 批量动作必须逐条校验数据权限。
- 单行对象动作优先放 `row_actions`，不要全部堆到页面 toolbar。

## 4. 展示模式

| 模式 | 说明 |
| --- | --- |
| `mega` | 顶部固定导航 + Mega menu 作为主要入口 |
| `sidebar` | 顶部固定导航 + 侧面导航作为主要业务导航 |
| `both` | 顶部固定导航 + Mega menu + 当前 app 侧面导航 |
| `compact` | 顶部固定导航 + 折叠侧栏或移动端抽屉 |

无论展示模式如何变化，菜单数据结构、变量职责和权限过滤必须一致。

桌面端可以把顶部固定导航、页内导航和 toolbar 放在同一视觉行，但 DOM 和变量必须分开渲染，并用分隔符、留白、颜色或边框区分逻辑区域。

移动端建议:

- 右上入口: 系统导航、应用、通知、用户菜单。
- 右下入口: 页内导航、当前页面快捷动作。
- 侧面导航收缩为抽屉，或由 Mega menu / 网站地图承担找路。

右上处理系统级导航，右下处理当前页面上下文，两者不得混用。

## 5. 超层级处理

超过层级上限时，不继续加深菜单，改用:

- 页内选项卡。
- 筛选器。
- 树形列表或树表。
- 表格分组。
- 折叠面板。
- 步骤条。
- 搜索。
- 拆分页面或拆分业务模块。

## 6. 存量治理顺序

1. 盘点顶部导航、Mega menu、侧栏、面包屑、页内导航、toolbar 和数据库菜单。
2. 统一顶部固定导航来源，清理 app 内重复顶部菜单。
3. 从顶部固定导航中移除普通业务菜单。
4. 建立 `response.global_menu`，用于 Mega menu 和网站地图。
5. 把当前 app 普通业务菜单迁到 `response.side_nav` 或 `response.sub_menu`。
6. 把当前页面内部视图迁到 `response.page_nav` 或 `response.page_menu`。
7. 把页面动作迁到 `response.toolbar` 或 `row_actions`。
8. 标记 iframe 历史入口状态: 保留、改造、归档、删除。
9. 清理无入口页面、重复入口和失效 URL。

## 7. 新 app 接入清单

- [ ] 已接入 `service_center` 统一认证。
- [ ] 顶部固定导航未混入普通业务菜单。
- [ ] 全局入口已登记到 `response.global_menu`。
- [ ] Mega menu / 网站地图最多两层，第一层分类头不跳转。
- [ ] 当前 app 业务导航写入 `response.side_nav` 或 `response.sub_menu`。
- [ ] 侧面导航最多两层。
- [ ] 当前页面内部导航写入 `response.page_nav` 或 `response.page_menu`。
- [ ] 页内导航最多两层。
- [ ] Toolbar 如存在，只放当前页面动作或强相关快捷跳转。
- [ ] 菜单入口和 toolbar 动作都有后端权限校验。
- [ ] 菜单 `key` 稳定且不重复。

## 8. 项目登记

每个项目在 `projects/<web2py_service>/菜单治理清单.md` 或项目级导航文档中登记:

- 顶部固定导航入口。
- Mega menu / 网站地图二层结构。
- 当前 app 侧面导航。
- 面包屑规则。
- 页内导航。
- Toolbar 使用位置。
- 页面路由和权限。
- 存量菜单清理状态。

如本文与历史设计文档冲突，以本文为准。
