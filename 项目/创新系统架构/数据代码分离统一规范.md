# 数据代码分离统一规范

> 版本: v2.4  
> 日期: 2026-06-23  
> 适用项目: web2py 服务，兼容云服务商 PaaS、自建 Docker/Compose 和传统 `/opt/<web2py_service>` 部署  
> 适用范围: 新 app、新功能、主动改造 app、历史软链接 app 的数据分离治理

本文是 web2py 服务平台“数据代码分离”的唯一当前执行文档。原来的运行数据目录配置化、质量检查应用数据代码分离、默认 web2py app 改造作业指导、生产数据分离方案示例都已合并到本文；历史版本只保留在 `90-历史参考/数据分离历史参考/` 作为追溯资料，不再作为执行依据。

核心目标是让 Git 只同步代码、配置模板和文档，真实业务数据、生产配置、运行临时文件统一落在 `/data/<web2py_service>/<app>` 这个逻辑运行数据路径，并逐步减少对 app 目录软链接的依赖。传统宿主机部署直接使用宿主机 `/data`；PaaS 或容器化部署必须把数据库、上传文件和运行配置映射到平台持久化存储。

## 0. 最小实现案例

新 app 或主动改造 app 先按本节完成最小闭环，再按后文补齐迁移、回滚、验收和历史兼容细节。

目标结果:

- Git 只提交 app 代码和 `private/appconfig.ini.example`。
- 真实配置放在 `/data/<web2py_service>/<app>/config/appconfig.ini`。
- SQLite 数据库写入 `/data/<web2py_service>/<app>/databases`。
- 上传附件写入 `/data/<web2py_service>/<app>/uploads`。

先创建运行目录:

```bash
mkdir -p /data/<web2py_service>/<app>/{config,databases,uploads,backups}
mkdir -p /data/<web2py_service>/<app>/runtime/{sessions,errors,cache,logs}
```

提交到 Git 的配置模板:

```text
/opt/<web2py_service>/applications/<app>/private/appconfig.ini.example
```

```ini
[storage]
runtime_root = /data/<web2py_service>/<app>

[db_<app>]
uri = sqlite://<app>.sqlite
migrate = true
pool_size = 10
```

生产环境真实配置:

```text
/data/<web2py_service>/<app>/config/appconfig.ini
```

`models/01_db.py` 最小写法:

```python
import os

from gluon.contrib.appconfig import AppConfig

APP_NAME = request.application
DEFAULT_RUNTIME_ROOT = os.path.join("/data/<web2py_service>", APP_NAME)
EXTERNAL_CONFIG = os.path.join(DEFAULT_RUNTIME_ROOT, "config", "appconfig.ini")

configuration = (
    AppConfig(EXTERNAL_CONFIG, reload=True)
    if os.path.exists(EXTERNAL_CONFIG)
    else AppConfig(reload=True)
)

runtime_root = configuration.get("storage.runtime_root") or DEFAULT_RUNTIME_ROOT
database_folder = (
    configuration.get("storage.database_folder")
    or os.path.join(runtime_root, "databases")
)
upload_root = (
    configuration.get("storage.upload_root")
    or os.path.join(runtime_root, "uploads")
)

for path in (database_folder, upload_root):
    if not os.path.exists(path):
        os.makedirs(path)

db_app = DAL(
    configuration.get("db_%s.uri" % APP_NAME) or "sqlite://%s.sqlite" % APP_NAME,
    migrate=True,
    pool_size=10,
    folder=database_folder,
)


def UploadField(name, *args, **kwargs):
    subdir = kwargs.pop("subdir", None)
    uploadfolder = os.path.join(upload_root, subdir) if subdir else upload_root
    if not os.path.exists(uploadfolder):
        os.makedirs(uploadfolder)
    kwargs.setdefault("uploadfolder", uploadfolder)
    return Field(name, "upload", *args, **kwargs)
```

业务表定义:

```python
db_app.define_table(
    "document",
    Field("title", "string", label="标题"),
    UploadField("attachment", label="附件"),
    UploadField("photo", label="现场照片", subdir="photos"),
)
```

下载接口:

```python
def download():
    return response.download(request, db_app)
```

这就是数据代码分离的最小闭环: DAL 的 `folder=` 决定数据库目录，upload 字段的 `uploadfolder=` 决定附件目录，生产配置和运行数据都不进入 Git。

## 1. 统一口径

- 代码目录在传统部署中通常为 `/opt/<web2py_service>/applications/<app>`；容器内为 web2py 根目录下的 `applications/<app>`。
- 运行数据逻辑目录固定为 `/data/<web2py_service>/<app>`；PaaS 或容器化部署必须有等价的持久化存储。
- 业务数据库通过 DAL `folder=` 指向外部数据目录。
- 上传文件通过 web2py 官方 `uploadfolder=` 指向外部上传目录。
- 真实生产配置不提交 Git，放在 `/data/<web2py_service>/<app>/config/appconfig.ini`。
- `sessions`、`errors`、`cache`、运行日志不进入 Git。
- 软链接只作为历史兼容或短期回滚手段，不作为新开发标准。
- 数据目录配置化是数据代码分离的实现机制，不再单独维护一份方案文档。
- 生产数据分离示例不再单独维护，项目实例和迁移做法统一收敛到本文。

## 2. 存储架构总览

cxmj 类内部系统按“代码、配置、业务数据、运行临时数据、备份”五类存储分层。新 app 和主动改造 app 必须按下表落位。

| 层级 | 标准位置 | 负责人 | Git 同步 | 备份 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 代码层 | `applications/<app>`，传统部署实例路径为 `/opt/<web2py_service>/applications/<app>` | 开发/发布流程 | 是 | 由 Git 管理 | models、controllers、modules、views、static、cron、配置模板 |
| 配置层 | `/data/<web2py_service>/<app>/config/appconfig.ini` | 运维/项目负责人 | 否 | 必须 | 生产数据库 URI、运行目录、密钥、第三方配置 |
| 业务数据层 | `/data/<web2py_service>/<app>/databases` | 应用运行用户 | 否 | 必须 | SQLite、迁移元数据、业务主数据 |
| 附件层 | `/data/<web2py_service>/<app>/uploads` | 应用运行用户 | 否 | 必须 | 图片、图纸、凭证、文档、导入原始文件 |
| 运行临时层 | `/data/<web2py_service>/<app>/runtime` | 应用运行用户/运维 | 否 | 按需 | sessions、errors、cache、logs |
| 备份层 | `/data/<web2py_service>/<app>/backups` 或统一备份目录 | 运维 | 否 | 异地/周期归档 | 迁移前快照、手工备份、恢复包 |

推荐运行时访问路径:

```text
web2py app
  ├─ 读取代码: <web2py_root>/applications/<app>
  ├─ 读取真实配置: /data/<web2py_service>/<app>/config/appconfig.ini
  ├─ DAL(folder=...): /data/<web2py_service>/<app>/databases
  ├─ Field(uploadfolder=...): /data/<web2py_service>/<app>/uploads[/<subdir>]
  └─ runtime: /data/<web2py_service>/<app>/runtime
```

关键边界:

- `/opt` 下 app 目录是发布产物，不是运行数据仓库。
- `/data` 下 app 目录是运行事实，不通过 Git 发布。
- PaaS 或容器化部署中 `/data` 必须映射到持久化存储，不得只存在于实例临时层。
- 代码通过配置读取 `/data`，不得靠硬编码旧项目路径。
- 软链接只用于历史兼容和回滚，不是新架构标准。
- 每个 app 的配置、数据库、上传和 runtime 默认放在同一个 `/data/<web2py_service>/<app>` 根下；确有独立磁盘需求时，通过配置显式覆盖。

## 3. 责任边界

| 对象 | 必须做 | 禁止做 |
| --- | --- | --- |
| 开发人员 | 提交配置模板、使用 DAL `folder=`、使用 `uploadfolder=` 或 `UploadField` 封装、提供迁移说明 | 提交真实配置、把上传写回 app 内、在控制器里硬编码 `/data` 散乱路径 |
| 运维人员 | 创建 `/data` 目录、设置属主权限、放置真实配置、执行备份恢复 | 直接修改 Git 管理的代码文件作为长期方案 |
| 项目负责人 | 在 `projects/<web2py_service>/` 记录项目实际路径、app 清单、数据目录状态 | 让项目级文档复制一整份通用规范 |
| 历史 app 维护者 | 保留兼容软链接时记录原因、观察期和清理计划 | 把软链接提交到 Git 或把软链接当作新开发标准 |

## 4. 数据分类

数据分离不是把 app 下所有默认目录都移动到 `/data`，而是先区分业务持久数据、框架运行数据、生产配置、代码资源和可再生产物。

| 类别 | 标准目录 | Git 同步 | 备份要求 | 说明 |
| --- | --- | --- | --- | --- |
| 业务数据库 | `/data/<web2py_service>/<app>/databases` | 否 | 必须备份 | SQLite、迁移元数据、业务主数据 |
| 上传附件 | `/data/<web2py_service>/<app>/uploads` | 否 | 必须备份 | 图片、图纸、凭证、文档 |
| 生产配置 | `/data/<web2py_service>/<app>/config/appconfig.ini` | 否 | 必须安全备份 | 不提交真实密钥和生产参数 |
| 会话 | `/data/<web2py_service>/<app>/runtime/sessions` | 否 | 默认不备份 | 可重新登录生成 |
| 缓存 | `/data/<web2py_service>/<app>/runtime/cache` | 否 | 默认不备份 | 可重新生成 |
| 错误票据 | `/data/<web2py_service>/<app>/runtime/errors` | 否 | 可选归档 | 排障期保留 |
| 日志 | `/data/<web2py_service>/<app>/runtime/logs` | 否 | 按运维策略 | 不进入代码仓库 |

web2py 默认 app 目录处理规则:

| app 默认目录 | 用途 | 是否业务数据 | 当前处理 |
| --- | --- | --- | --- |
| `databases/` | SQLite 数据库、DAL 表结构元数据、迁移记录、`sql.log` | 是 | 必须重定位到 `/data/.../databases` |
| `uploads/` | `Field(..., "upload")` 默认附件目录 | 是 | 必须通过 `uploadfolder=` 重定位到 `/data/.../uploads` |
| `private/` | 私有配置、密钥、模板和内部文件 | 配置相关 | 只提交模板，真实配置外置到 `/data/.../config/appconfig.ini` |
| `sessions/` | 文件型 session | 框架运行数据 | 不作为业务迁移前置项，可外置或按会话策略清理 |
| `errors/` | web2py ticket 错误文件 | 运维诊断数据 | 可外置、归档或定期清理，不阻塞业务数据分离 |
| `cache/` | 文件缓存 | 临时数据 | 可外置或清理，不纳入业务备份 |
| `languages/` | 翻译文件，开发时可能被 web2py 更新 | 代码/构建产物 | 生产环境按代码资源管理，尽量只读 |
| `models/`、`controllers/`、`modules/`、`views/`、`static/`、`cron/` | app 代码、模板、静态资源和任务定义 | 否 | 保留在代码目录，通过代码发布维护 |

## 5. 标准目录

代码目录:

```text
/opt/<web2py_service>/applications/<app>/
├── controllers/
├── models/
├── modules/
├── views/
├── static/
├── cron/
├── languages/
├── private/
│   └── appconfig.ini.example
└── README.md
```

运行数据目录:

```text
/data/<web2py_service>/<app>/
├── config/
│   └── appconfig.ini
├── databases/
├── uploads/
├── runtime/
│   ├── sessions/
│   ├── errors/
│   ├── cache/
│   └── logs/
└── backups/
```

目录创建规则:

```bash
mkdir -p /data/<web2py_service>/<app>/{config,databases,uploads,backups}
mkdir -p /data/<web2py_service>/<app>/runtime/{sessions,errors,cache,logs}
```

`photos`、`drawings`、`attachments`、`vouchers` 等上传子目录由 app 业务函数派生，不要求在标准目录层固定枚举。cron、导出任务或批处理脚本如果生成业务文件，输出目录必须显式写到 `/data/<web2py_service>/<app>` 下。

## 6. 配置化规则

每个新 app 或主动改造 app 必须提供配置模板:

```text
/opt/<web2py_service>/applications/<app>/private/appconfig.ini.example
```

真实配置放在:

```text
/data/<web2py_service>/<app>/config/appconfig.ini
```

生产环境最小配置:

```ini
[storage]
runtime_root = /data/<web2py_service>/<app>

[db_<app>]
uri = sqlite://<app>.sqlite
migrate = true
pool_size = 10
```

推荐完整配置项:

```ini
[storage]
runtime_root = /data/<web2py_service>/<app>
database_folder = /data/<web2py_service>/<app>/databases
upload_root = /data/<web2py_service>/<app>/uploads

[db_<app>]
uri = sqlite://<app>.sqlite
migrate = true
pool_size = 10

[session]
masterapp = service_center
```

配置读取优先级:

1. 环境变量，用于容器、临时迁移和生产覆盖。
2. `/data/<web2py_service>/<app>/config/appconfig.ini`，用于生产稳定配置。
3. `applications/<app>/private/appconfig.ini`，仅用于本地开发或兼容旧部署。
4. 代码内安全默认值，生产环境不得依赖默认值。

`storage.runtime_root` 是运行数据根目录。`database_folder` 和 `upload_root` 可以由它派生，也可以在需要独立磁盘或历史兼容时显式覆盖。

`storage.upload_root` 是本项目自定义配置项，不是 web2py 内置配置项。它用于声明 app 的默认外部上传根目录，由 app 配置模块或 `platform_storage.py` 读取，最终传给 web2py upload 字段的 `uploadfolder=`。

默认只配置 `runtime_root` 和必要时的 `upload_root`。通用上传可以直接落到上传根目录；`photos`、`drawings`、`attachments`、`vouchers` 等子目录由 app 内函数派生，不默认单独配置。确实需要独立磁盘时，再增加专用覆盖项:

```ini
[storage]
upload_root = /data/<web2py_service>/<app>/uploads
photo_folder = /mnt/quality-photo-storage
```

## 7. app 实现标准

### 5.1 共享路径工具

统一路径工具:

```text
/opt/<web2py_service>/applications/service_center/modules/platform_storage.py
```

app 配置模块应基于共享工具导出:

```python
runtime_config.upload_root
runtime_config.upload_path("photos")
runtime_config.upload_path("attachments")
```

模型初始化后应导出 app 内公共入口:

```python
APP_UPLOAD_ROOT = runtime_config.upload_root
APP_UPLOAD_PATH = runtime_config.upload_path
```

同一 app 内的控制器、模型和服务模块必须复用这些公共入口，不得各自拼接 `request.folder/uploads`。

### 5.2 数据库目录

数据库必须通过 DAL `folder=` 指向外部目录:

```python
db_app = DAL(
    runtime_config.db_uri,
    migrate=runtime_config.db_migrate,
    pool_size=runtime_config.db_pool_size,
    folder=runtime_config.database_folder,
)
```

禁止依赖:

```text
/opt/<web2py_service>/applications/<app>/databases
```

历史 app 如果仍依赖该目录，可以临时保留软链接，但软链接不得进入 Git。

### 5.3 上传目录

app 必须提供一个默认外部上传根目录，业务表通用上传可以统一使用该目录:

```python
APP_UPLOAD_ROOT = runtime_config.upload_root

Field("file", "upload", uploadfolder=APP_UPLOAD_ROOT)
```

有明确业务类别的上传再使用派生子目录:

```python
Field("site_photo", "upload", uploadfolder=APP_UPLOAD_PATH("photos"))
Field("attachment", "upload", uploadfolder=APP_UPLOAD_PATH("attachments"))
```

为了避免每个字段重复写 `uploadfolder=...`，app 可以在模型启动层提供统一封装:

```python
def UploadField(name, *args, **kwargs):
    subdir = kwargs.pop("subdir", None)
    kwargs.setdefault(
        "uploadfolder",
        APP_UPLOAD_PATH(subdir) if subdir else APP_UPLOAD_ROOT,
    )
    return Field(name, "upload", *args, **kwargs)

UploadField("file", label="附件")
UploadField("site_photo", label="现场照片", subdir="photos")
```

封装函数必须在 `models/01_db.py` 或 app 的公共模型中定义，后续业务表统一使用它。这样可以做到“默认外部 upload 根目录”只配置一次，同时仍避免裸 upload 字段落回 app 内目录。

参考案例:

`private/appconfig.ini.example`:

```ini
[storage]
runtime_root = /data/<web2py_service>/sample_app
upload_root = /data/<web2py_service>/sample_app/uploads

[db_sample_app]
uri = sqlite://sample_app.sqlite
migrate = true
pool_size = 10
```

`models/01_db.py`:

```python
import os
import sys

from gluon.contrib.appconfig import AppConfig

shared_modules = os.path.join(request.folder, "..", "service_center", "modules")
if shared_modules not in sys.path:
    sys.path.append(shared_modules)

from platform_storage import ensure_directory, upload_path, upload_root

configuration = AppConfig(reload=True)

APP_UPLOAD_ROOT = ensure_directory(
    upload_root(
        request,
        configuration=configuration,
        app_name=request.application,
    )
)


def APP_UPLOAD_PATH(subdir=None):
    return ensure_directory(
        upload_path(
            request,
            subpath=subdir,
            configuration=configuration,
            app_name=request.application,
            upload_root_path=APP_UPLOAD_ROOT,
        )
    )


def UploadField(name, *args, **kwargs):
    subdir = kwargs.pop("subdir", None)
    kwargs.setdefault(
        "uploadfolder",
        APP_UPLOAD_PATH(subdir) if subdir else APP_UPLOAD_ROOT,
    )
    return Field(name, "upload", *args, **kwargs)
```

`models/20_tables.py`:

```python
db.define_table(
    "sample_document",
    Field("title", "string", label="标题"),
    UploadField("attachment", label="附件"),
    UploadField("photo", label="现场照片", subdir="photos"),
    Field("created_on", "datetime", default=request.now),
)
```

`controllers/default.py`:

```python
def upload_demo():
    form = SQLFORM(db.sample_document)
    if form.process().accepted:
        redirect(URL("upload_demo"))
    return dict(form=form, rows=db(db.sample_document).select(orderby=~db.sample_document.id))


def download():
    return response.download(request, db)
```

此案例中 `storage.upload_root` 是项目自定义配置。web2py 实际使用的是 `UploadField` 内部设置的 `uploadfolder=APP_UPLOAD_ROOT` 或 `uploadfolder=APP_UPLOAD_PATH("photos")`。

禁止写法:

```python
Field("file", "upload")
Field("file", "upload", uploadfolder=os.path.join(request.folder, "uploads"))
```

说明:

- 裸 `Field("file", "upload")` 会默认写入 app 内 `uploads/`，不符合数据分离要求；除非该 app 已统一改用上面的 `UploadField` 封装，否则不得使用裸 upload 字段。
- `uploadfolder=` 是 web2py 官方支持的上传目录指定方式。
- Python 3.10 下，web2py 上传对象可能是 `cgi.FieldStorage`，不能写 `if upload`、`if not upload`、`bool(upload)` 或 `if filename and fileobj`；必须用 `upload is None`、`filename` 和 `fileobj is None` 显式判断。
- `uploadseparate=True` 可以用于大量文件分散子目录，但上线后不得随意切换。
- 不建议使用 `uploadfield` 把大附件写入数据库。当前标准是数据库保存文件名，文件实体放到 `/data/<web2py_service>/<app>/uploads`。

### 5.4 临时目录

`sessions`、`errors`、`cache` 不作为核心业务数据保存。启动层或部署脚本负责创建目录、检查权限和按策略清理。

处理原则:

- `sessions` 可清理，清理后用户重新登录。
- `cache` 可清理，清理后重新生成。
- `errors` 保留近期 ticket 供排障，过期后归档或清理。
- 这些目录不应进入 Git，也不应阻塞代码发布。

## 8. 新 app 作业流程

1. 创建 `/opt/<web2py_service>/applications/<app>` 代码目录。
2. 创建 `/data/<web2py_service>/<app>/{config,databases,uploads,runtime,backups}`。
3. 提交 `private/appconfig.ini.example`，不要提交真实 `appconfig.ini`。
4. 在 `/data/<web2py_service>/<app>/config/appconfig.ini` 写入真实配置。
5. 在 app 的配置模块中读取外部配置和环境变量。
6. 在 `models/01_db.py` 或等价启动模型中通过 `folder=` 设置数据库目录。
7. 在模型初始化后导出 `APP_UPLOAD_ROOT` 和 `APP_UPLOAD_PATH`。
8. 通用 upload 字段使用 `uploadfolder=APP_UPLOAD_ROOT` 或统一 `UploadField` 封装；有明确业务类别的上传再使用 `APP_UPLOAD_PATH("<subdir>")`。
9. 接入 `service_center` 统一认证，不新建独立认证库。
10. 执行 CI、运行目录审计和页面验证。

## 10. 旧 app 改造流程

### 8.1 改造前盘点

```bash
find /opt/<web2py_service>/applications/<app> -maxdepth 2 \( -name databases -o -name uploads -o -name sessions -o -name errors -o -name cache \) -ls
du -sh /opt/<web2py_service>/applications/<app>/databases /opt/<web2py_service>/applications/<app>/uploads 2>/dev/null || true
rg -n "uploadfolder|Field\\(.*['\"]upload['\"]|request\\.folder.*uploads" /opt/<web2py_service>/applications/<app> -g '*.py'
```

重点确认:

- 当前数据库文件和 `.table` 迁移元数据位置。
- 上传字段是否依赖默认 `uploads/`。
- 是否存在独立 `Auth` 和独立 `auth_user`。
- 是否已有软链接以及软链接是否断链。

### 8.2 备份旧数据

```bash
tar -czf /data/<web2py_service>/<app>/backups/pre-separation-$(date +%Y%m%d-%H%M%S).tar.gz \
  -C /opt/<web2py_service>/applications <app>
```

如果旧数据目录较大，应优先单独备份 `databases/` 和 `uploads/`，并记录校验值。

### 8.3 迁移核心数据

建议迁移前停止 web2py 服务，避免 SQLite 和上传文件在复制过程中继续写入。

```bash
mkdir -p /data/<web2py_service>/<app>/{config,databases,uploads,backups}
mkdir -p /data/<web2py_service>/<app>/runtime/{sessions,errors,cache,logs}
rsync -a /opt/<web2py_service>/applications/<app>/databases/ /data/<web2py_service>/<app>/databases/
rsync -a /opt/<web2py_service>/applications/<app>/uploads/ /data/<web2py_service>/<app>/uploads/
```

只迁移核心数据。`sessions`、`errors`、`cache` 按临时目录处理，不作为必须完整迁移项。

迁移后不要立即删除旧目录，先改名保留观察期:

```bash
mv /opt/<web2py_service>/applications/<app>/databases \
  /opt/<web2py_service>/applications/<app>/databases.disabled

mv /opt/<web2py_service>/applications/<app>/uploads \
  /opt/<web2py_service>/applications/<app>/uploads.disabled
```

观察期确认新目录持续正常读写后再清理旧目录。

### 8.4 改造默认 web2py app

默认脚手架 app 推荐整理为:

```text
models/
  00_app.py
  01_db.py
  20_tables.py
  30_menu.py
  40_permissions.py
  README.md
```

文件职责:

| 文件 | 职责 |
| --- | --- |
| `00_app.py` | app 名称、页面元数据、静态版本号 |
| `01_db.py` | 统一认证、配置读取、业务库连接、公共路径变量 |
| `20_tables.py` | app 私有业务表，不创建认证表 |
| `30_menu.py` | 平台顶部菜单接入和 app 内菜单 |
| `40_permissions.py` | app 角色初始化和权限判断 |

改造要求:

- 业务表定义在 app 业务库上，例如 `_db.define_table(...)` 或 `_app_db.define_table(...)`。
- 认证用户继续来自 `service_center` 的 `db.auth_user`。
- 上传字段使用 app 公共上传目录。
- 下载接口传入业务库，例如 `response.download(request, _db)`。
- 用户登录、注册、资料维护跳转到 `service_center` 统一入口。
- 路径和数据库连接集中在 `01_db.py` 或配置模块，不散落到控制器。

## 11. 项目实例写法

`<quality_app>` 是当前较新的参考实现，核心做法是“显式目录重定位 + 认证/业务数据分库”。

边界:

- `service_center` 负责统一认证、用户、角色、会话入口和平台顶部导航。
- `<quality_app>` 负责质量检查业务代码、业务表、业务附件和业务页面。
- 认证数据和业务数据使用不同数据库连接。
- 业务数据目录由配置解析，不依赖软链接作为主路径。

关键变量:

```text
db             统一认证库
db_app  质量检查业务库
_app_db 质量检查业务库本地别名
APP_DB_DIR     质量检查业务数据库目录
APP_UPLOAD_DIR 质量检查上传目录
```

质量检查业务表必须定义在 `db_app` 或 `_app_db` 上，不能定义在认证库 `db` 上。引用认证用户时允许使用:

```python
IS_IN_DB(db, "auth_user.id", "%(first_name)s")
```

下载和预览必须使用业务库:

```python
def download():
    return response.download(request, _app_db)
```

项目实例文档如需记录差异，只写项目特有内容:

```text
# <app> 数据代码分离实施记录

- 统一规范: 数据代码分离统一规范.md
- 业务数据库: /data/<web2py_service>/<app>/databases/<app>.sqlite
- 上传目录: /data/<web2py_service>/<app>/uploads
- 配置文件: /data/<web2py_service>/<app>/config/appconfig.ini
- 已完成迁移时间: YYYY-MM-DD HH:MM
- 已验证页面: ...
- 保留观察的旧目录: ...
```

不要再复制一份通用生产数据分离方案，避免多个文档同时成为执行口径。

## 13. 验证标准

发布前执行:

```bash
cd /opt/<web2py_service>
python3 -m py_compile applications/service_center/modules/platform_storage.py
scripts/ci_check.sh
scripts/runtime_links.sh audit --strict
```

重点检查:

```bash
git ls-files | grep -E 'applications/.+/(databases|uploads|sessions|errors|cache)(/|$)' && echo "BAD: runtime path tracked"
rg -n 'request\.folder.*uploads|uploads.*request\.folder' applications/<app>
rg -n "Field\(.*['\"]upload['\"]" applications/<app>
```

功能验证:

- 新增一条业务记录。
- 上传一张测试附件。
- 确认数据库文件写入 `/data/<web2py_service>/<app>/databases`。
- 确认附件写入 `/data/<web2py_service>/<app>/uploads/<subdir>`。
- 确认 `/opt/<web2py_service>/applications/<app>/uploads` 没有新增业务文件。
- 页面下载和预览功能正常。
- 清理 session 后可重新登录。

可选目录权限检查:

```bash
test -w /data/<web2py_service>/<app>/databases
test -w /data/<web2py_service>/<app>/uploads
test -w /data/<web2py_service>/<app>/runtime/cache
test -w /data/<web2py_service>/<app>/runtime/errors
test -w /data/<web2py_service>/<app>/runtime/logs
test -w /data/<web2py_service>/<app>/runtime/sessions
```

推荐增加只读诊断页，展示:

- 当前配置文件路径。
- `runtime_root`。
- 数据库目录、上传目录、备份目录。
- 目录是否存在、是否可写、是否为软链接。
- 目录文件数量和最近上传文件时间。
- app 内默认 `databases/` 和 `uploads/` 是否仍有新文件产生。

诊断页只读，不直接修改数据库目录。路径修改应通过配置文件和服务重启完成，不在页面中热切换 DAL。

## 14. 回滚原则

回滚只回滚代码时:

- 保留 `/data/<web2py_service>/<app>`。
- 不删除已迁移数据。
- 如旧代码仍依赖 app 内路径，可临时使用软链接恢复访问。

临时软链接回滚示例:

```bash
ln -s /data/<web2py_service>/<app>/databases /opt/<web2py_service>/applications/<app>/databases
ln -s /data/<web2py_service>/<app>/uploads /opt/<web2py_service>/applications/<app>/uploads
```

回滚后必须重新执行:

```bash
scripts/runtime_links.sh audit --strict
scripts/ci_check.sh
```
