# 软件系统规范化设计执行清单

本文档用于指导 web2py 服务平台从“直接浏览网站理解系统”逐步过渡到“通过系统地图、菜单清单、页面清单、权限矩阵和截图索引掌控全局”的规范化设计开发方式。

## 1. 总体原则

规范化设计不是一次性把所有页面和细节画死，而是先建立稳定骨架，再在统一规则下持续扩展。

核心原则：

- 先定系统边界，再定菜单和页面。
- 先定业务能力地图，再定具体路由和实现。
- 平台能力集中，业务能力分散。
- 菜单渲染集中，菜单内容按归属声明。
- 顶部导航、左侧导航、页内导航、页面工具栏职责分离。
- 每次新增功能必须同步更新菜单、页面、权限、截图和变更记录。

推荐导航分层：

```text
顶部导航：系统级、组织级、用户级入口
左侧导航：平台级 / 模块级入口，最多两级
页内导航：当前 app 或当前业务域内部页面
页面工具栏：新增、编辑、导入、导出、审批、刷新等动作
面包屑：当前位置说明
```

## 2. 阶段一：建立系统总览

目标：先形成“系统地图”，减少只靠浏览网站理解系统的风险。

执行清单：

- [x] 新建或维护 `applications/docs/00-总览/系统导航与页面总览.md`。
- [x] 列出顶部导航、平台左侧导航、各 app 页内导航。
- [x] 建立页面路由清单。
- [x] 建立核心页面截图索引。
- [x] 标记历史菜单、重复 layout、重复 CSS、未使用页面。
- [x] 标记每个菜单项的状态：保留、合并、迁移、删除、待确认。

建议总览文档结构：

```markdown
# 系统导航与页面总览

## 1. 平台能力地图
## 2. 顶部菜单
## 3. 平台左侧菜单
## 4. 各 app 页内菜单
## 5. 页面路由清单
## 6. 权限矩阵
## 7. 页面截图索引
## 8. 历史菜单残留清理清单
## 9. 待确认问题
```

页面清单字段建议：

| 字段 | 说明 |
| --- | --- |
| 页面名称 | 用户看到的页面名称 |
| 所属 app | 例如 `service_center`、`<business_app>`、`<quality_app>` |
| 菜单路径 | 顶部 / 左侧 / 页内菜单路径 |
| URL | 实际访问地址 |
| 控制器 | web2py controller |
| 方法 | web2py function |
| 页面用途 | 业务目的 |
| 权限码 | 菜单或页面权限 |
| 数据来源 | 主要表或接口 |
| 截图路径 | 对应截图文件 |
| 状态 | 保留、合并、迁移、删除、待确认 |

## 3. 阶段二：定义信息架构

目标：先看业务能力，再决定菜单和页面，不从单个页面倒推整个系统。

执行清单：

- [ ] 绘制业务能力地图。
- [ ] 明确每个能力属于平台层、业务模块层还是页面操作层。
- [ ] 给每个菜单项判断归属层级。
- [ ] 将页面动作从菜单中移出，放到页面工具栏。
- [ ] 将三级及以下业务入口移到页内菜单。
- [ ] 确认左侧菜单最多两级。

业务能力地图示例：

```text
web2py 服务平台
  平台基础
    用户管理
    权限管理
    菜单管理
    系统配置

  业务系统 业务系统
    业务系统 看板
    产品档案
    作业指导书
    产品质量检查
    业务报表

  质量检查
    质量检查看板
    质量检查采集
    异常记录
    统计分析
    基础配置
```

菜单归属判断：

| 问题 | 应放位置 |
| --- | --- |
| 是否是系统级入口？ | 顶部导航 |
| 是否是业务域或模块入口？ | 平台左侧导航 |
| 是否只在当前 app 内使用？ | 页内导航 |
| 是否是新增、编辑、导入、导出等动作？ | 页面工具栏 |
| 是否只是当前位置说明？ | 面包屑 |

## 4. 阶段三：建立低保真设计规范

目标：不依赖完整视觉设计稿，也能统一页面结构。

当前阶段不强制引入 Figma。可以先用 Markdown、Mermaid、Excalidraw 或简单线框图建立低保真设计。

基础页面骨架：

```text
顶部导航
左侧导航
页面标题
页内菜单
筛选区 / 查询区
页面工具栏
主内容区
分页 / 状态区
```

需要定义的页面类型：

- [ ] 看板页
- [ ] 列表页
- [ ] 表单页
- [ ] 详情页
- [ ] 配置页
- [ ] 管理页
- [ ] 报表页

列表页标准结构：

```text
页面标题
页内菜单
筛选条件
工具栏：新增、导入、导出、刷新、批量操作
数据表格
分页
空状态 / 加载状态 / 错误状态
```

表单页标准结构：

```text
页面标题
面包屑
表单分组
校验提示
底部或顶部操作区：保存、取消、提交
```

看板页标准结构：

```text
页面标题
页内菜单或筛选条
核心指标
图表区域
明细区域
刷新时间和数据口径说明
```

## 5. 阶段四：代码治理

目标：让代码结构服从系统分层，减少重复实现。

执行清单：

- [ ] `service_center` 统一维护顶部导航、平台左侧导航、菜单 CSS、菜单 JS。
- [ ] 各业务 app 只声明自己的页内菜单。
- [ ] 各业务 app 不得覆盖 `response.sub_menu` 为 app 内部菜单。
- [ ] 各业务 app 不得复制左侧菜单 DOM。
- [ ] 各业务 app 不得覆盖 `.main-sidebar`、`.nav-sidebar`、`.nav-treeview`、`.brand-link` 等平台菜单通用样式。
- [ ] 清理旧变量，例如历史 app 中的 `xian_sub_menu`、自定义侧栏菜单等。
- [ ] 抽取或统一重复 layout 中的顶部菜单、用户菜单、资源引用逻辑。
- [ ] 新 app 必须复用统一布局、统一菜单渲染函数和统一 CSS/JS。
- [ ] 新 app 和主动改造 app 默认只配置 `storage.upload_root`，通过 `service_center/modules/platform_storage.py` 解析真实目录，并在 app 内导出公共 `*_UPLOAD_ROOT` / `*_UPLOAD_PATH`；子目录使用 `*_UPLOAD_PATH("photos")` 这类函数派生，不得在控制器、模型或服务模块中重新拼接 `request.folder/uploads`。

推荐代码归属：

```text
service_center
  modules/platform_nav.py      # 顶部菜单、平台左侧菜单数据
  modules/platform_ui.py       # 菜单和公共 UI 渲染
  modules/platform_storage.py  # 运行数据、上传目录解析
  static/css/adminlte-sidebar.css
  static/js/sidebar-state.js
  models/xx_menu.py            # 注入 response.menu / response.sub_menu

业务 app
  models/*_menu.py             # 只定义 response.page_menu
  controllers/*.py             # 页面控制器
  views/*.html                 # 页面内容
```

新增菜单时必须登记：

| 字段 | 要求 |
| --- | --- |
| 菜单名称 | 用户可理解的业务名称 |
| 所属层级 | 顶部、左侧、页内、工具栏 |
| 所属 app | 实际归属应用 |
| URL | 目标页面 |
| 权限码 | 菜单和页面权限 |
| 是否核心入口 | 是 / 否 |
| 是否需要截图 | 是 / 否 |
| 状态 | 新增、调整、废弃 |

## 6. 阶段五：截图和验收

目标：把页面视觉状态变成可追踪资产。

截图目录建议：

```text
applications/docs/screenshots/
  service_center/
  <business_app>/
  <quality_app>/
  <report_app>/
```

截图命名建议：

```text
app-controller-function-desktop.png
app-controller-function-mobile.png
```

每个核心页面至少记录：

- [ ] 菜单路径
- [ ] URL
- [ ] 控制器 / 方法
- [ ] 页面用途
- [ ] 主要操作
- [ ] 权限要求
- [ ] 数据来源
- [ ] 桌面截图
- [ ] 移动端截图

页面验收清单：

- [ ] 顶部导航高亮正确。
- [ ] 左侧菜单稳定，不被 app 页面替换。
- [ ] 左侧菜单展开和折叠状态符合预期。
- [ ] 页内菜单内容只包含当前业务域内部入口。
- [ ] 页内菜单高亮正确。
- [ ] 页面标题和面包屑正确。
- [ ] 权限隐藏后布局不破坏。
- [ ] 表格、表单、按钮可正常使用。
- [ ] 小屏幕不重叠、不溢出。
- [ ] 日夜模式下菜单和页面可读。

后续可用 Playwright 自动截图，形成视觉回归检查。

## 7. 阶段六：持续治理

目标：让规范成为日常流程，而不是一次性整理。

每次新增功能的标准流程：

```text
确认业务归属
  -> 判断菜单层级
  -> 更新菜单清单
  -> 更新页面路由清单
  -> 定义权限码
  -> 开发页面
  -> 补充截图
  -> 更新变更记录
  -> 验收菜单、权限、页面状态
```

每月架构质量检查：

- [ ] 是否出现重复菜单。
- [ ] 是否出现 app 覆盖平台左侧菜单。
- [ ] 是否出现重复 CSS。
- [ ] 是否出现重复 layout。
- [ ] 是否存在废弃页面。
- [ ] 是否存在未登记权限。
- [ ] 是否存在没有截图的核心页面。
- [ ] 是否存在菜单和页面不一致的情况。

## 8. 设计工具采用路线

当前阶段优先使用轻量工具，不需要一开始就引入完整设计平台。

建议顺序：

```text
Markdown 表格
Mermaid 架构图
浏览器截图
Excalidraw 低保真图
Figma 视觉规范和组件库
Playwright 自动截图
Storybook 或组件文档
```

工具使用建议：

| 阶段 | 工具 | 目的 |
| --- | --- | --- |
| 当前 | Markdown | 菜单、页面、权限、截图清单 |
| 当前 | Mermaid | 系统能力图、模块关系图 |
| 当前 | 浏览器截图 | 保存真实页面状态 |
| 当前可选 | Figma | 页面设计稿映射、视觉规范、组件库、设计评审 |
| 中期 | Excalidraw | 快速画低保真结构 |
| 中期 | Playwright | 自动截图和回归检查 |
| 后期 | Storybook | 组件级展示和测试 |

## 9. 最先执行的 5 件事

优先级最高的落地动作：

- [x] 生成 `applications/docs/00-总览/系统导航与页面总览.md`。
- [x] 列出当前所有顶部、左侧、页内菜单。
- [x] 建立页面路由清单。
- [x] 标记旧菜单、重复 layout、重复 CSS。
- [x] 选择 5 个核心页面建立截图索引。

完成以上 5 件事后，系统管理方式应从“打开网站逐页看”转为“通过系统地图总览全局，再进入页面验证细节”。
