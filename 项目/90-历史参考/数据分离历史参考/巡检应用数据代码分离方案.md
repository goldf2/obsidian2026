# <quality_app> 数据代码分离方案

版本：v1.0  
日期：2026-06-14  
适用应用：`applications/<quality_app>`  
参考实现：`service_center` 统一认证体系 + `<quality_app>` 独立业务库

## 1. 方案定位

`<quality_app>` 当前采用的是平台内较新的应用分离方案。它的核心是“显式目录重定位 + 认证/业务数据分库”，而不是依赖软链接来搬运目录。

- `service_center` 负责统一认证、用户、角色、会话入口和平台顶部导航。
- `<quality_app>` 只负责质量检查业务代码、业务表、业务附件和业务页面。
- 认证数据和业务数据使用不同数据库连接，避免业务表污染统一用户库。
- 运行期数据目录继续遵循 web2py 标准目录名，但真实位置由配置清单显式声明，便于系统迁移时审计和复现。

这套方案适合作为其他业务 app 接入统一平台时的参考模板。

## 2. 当前落地形态

### 2.1 代码目录

`<quality_app>` 的代码主体位于：

```text
applications/<quality_app>/
  controllers/
  models/
  views/
  static/
  private/
```

关键代码文件：

- `models/00_app.py`：应用名称、页面元数据。
- `models/01_db.py`：接入 `service_center` 统一认证，并创建 `db_app` 业务数据库。
- `models/20_tables.py`：定义质量检查业务表，统一使用 `_app_db`。
- `models/30_menu.py`：加载 `service_center` 的统一顶部菜单。
- `models/40_permissions.py`：初始化质量检查角色并提供权限判断函数。
- `controllers/default.py`：质量检查页面、上传、下载、业务操作入口。

### 2.2 运行数据目录

`<quality_app>` 仍保留 web2py 标准运行目录语义：

```text
applications/<quality_app>/
  databases/
  uploads/
  sessions/
  errors/
  cache/
```

推荐部署形态：

- 开发环境：可以直接使用 app 内部标准目录，也可以使用 `runtime_data/...` 这类本地运行目录。
- 生产环境：使用 `private/appconfig.ini` 或环境变量显式指定各数据目录，不依赖软链接。

推荐配置清单：

```ini
[runtime]
root               = /data/<web2py_service>/<quality_app>
databases_dir      = /data/<web2py_service>/<quality_app>/databases
uploads_dir        = /data/<web2py_service>/<quality_app>/uploads
sessions_dir       = /data/<web2py_service>/<quality_app>/sessions
errors_dir         = /data/<web2py_service>/<quality_app>/errors
cache_dir          = /data/<web2py_service>/<quality_app>/cache
auth_root          = /data/<web2py_service>/service_center
auth_databases_dir = /data/<web2py_service>/service_center/databases
```

代码层只依赖 `01_db.py` 暴露的 `APP_DB_DIR`、`APP_UPLOAD_DIR` 等变量。软链接只作为历史兼容方式，诊断页可以识别，但新迁移不建议把软链接作为主方案。

### 2.3 Git 边界

运行数据目录不应进入 Git：

```text
applications/*/databases/
applications/*/uploads/
applications/*/sessions/
applications/*/errors/
applications/*/cache/
applications/*/private/appconfig.ini
```

应进入 Git 的内容：

- `controllers/`
- `models/`
- `views/`
- `static/`
- `private/appconfig.ini.example`
- 应用说明文档和部署脚本

真实的 `private/appconfig.ini` 属于本机或生产环境配置，不作为代码发布内容。

## 3. 总体架构

```text
用户浏览器
  |
  v
<quality_app> 页面入口
  |
  +-- 统一登录 / 会话 / 用户角色
  |     |
  |     v
  |   service_center
  |     |
  |     v
  |   auth_root/databases/storage.sqlite
  |     auth_user
  |     auth_group
  |     auth_membership
  |
  +-- 质量检查业务数据
        |
        v
      databases_dir/db_app.sqlite
        inspection_product
        inspection_process
        inspection_process_page
        inspection_checkpoint
        inspection_record
        inspection_record_photo
```

其中 `databases_dir` 来自 `<quality_app>` 的 `[runtime] databases_dir`，未配置时才从 `[runtime] root` 推导；`auth_root` 和 `auth_databases_dir` 来自 `[runtime]` 中的认证目录配置。本地参考配置目前指向 `runtime_data/<web2py_service>/...`，生产环境应指向独立数据盘目录。

数据边界：

- `db`：统一认证库，由 `service_center_auth.apply_service_center_auth()` 创建。
- `db_app`：质量检查业务库，由 `<quality_app>/models/01_db.py` 创建。
- `_app_db`：`db_app` 的本地别名，避免和认证库 `db` 混用。

## 4. 认证和会话分离

### 4.1 统一认证入口

`<quality_app>/models/01_db.py` 将 `service_center/modules` 加入 `sys.path`，然后调用：

```python
from service_center_auth import apply_service_center_auth
apply_service_center_auth(globals())
```

调用完成后，当前应用上下文中会获得：

- `db`：认证数据库连接。
- `auth`：统一认证对象。
- `crud`：认证库 CRUD 对象。
- `session`：按 `session.masterapp` 连接后的会话。

### 4.2 会话归属

`<quality_app>/private/appconfig.ini` 中配置：

```ini
[session]
masterapp = service_center
```

`service_center_auth` 会读取该配置并执行：

```python
session.connect(request, response, masterapp=masterapp)
```

效果：

- 登录状态由 `service_center` 统一管理。
- 用户访问 `<quality_app>` 时复用平台登录态。
- `<quality_app>` 不再维护独立用户体系。

### 4.3 用户入口处理

`<quality_app>/controllers/default.py` 中的 `user()` 不再提供独立用户管理，而是转发到统一登录入口：

```python
redirect(URL('service_center', 'default', 'user', args=request.args))
```

这样可以保证用户注册、登录、找回密码、资料维护等行为都走统一平台。

## 5. 数据库分离设计

### 5.1 认证数据库

认证数据库由 `service_center_auth` 创建：

```python
db = DAL(
    'sqlite://storage.sqlite',
    folder=os.path.join(os.path.dirname(request.folder), 'service_center', 'databases'),
    ...
)
```

职责：

- 用户账号：`auth_user`
- 角色：`auth_group`
- 用户角色关系：`auth_membership`
- 权限、事件、调度等平台级表

`<quality_app>` 可以查询认证库中的用户和角色，但不在认证库里创建质量检查业务表。

### 5.2 业务数据库

质量检查业务数据库由 `<quality_app>/models/01_db.py` 创建：

```python
db_app = DAL(
    DB_<QUALITY_APP>_URI,
    migrate=DB_<QUALITY_APP>_MIGRATE,
    pool_size=DB_<QUALITY_APP>_POOL_SIZE,
    folder=APP_DB_DIR,
)
```

职责：

- 适用件：`inspection_product`
- 质量作业指导书：`inspection_process`
- 指导书分页：`inspection_process_page`
- 检查点：`inspection_checkpoint`
- 检测记录：`inspection_record`
- 检测照片：`inspection_record_photo`

### 5.3 代码约束

业务表定义统一使用：

```python
_app_db = db_app
_app_db.define_table(...)
```

跨库引用规则：

- 引用认证用户时使用 `db.auth_user` 或 `IS_IN_DB(db, "auth_user.id", ...)`。
- 引用质量检查业务表时使用 `_app_db.inspection_*`。
- 不在 `db` 上定义 `inspection_*` 表。
- 不在 `_app_db` 上定义 `auth_*` 表。

示例：

```python
_app_db.inspection_record.inspector_id.requires = IS_EMPTY_OR(
    IS_IN_DB(db, "auth_user.id", "%(first_name)s")
)
```

这种写法表达的是“业务记录保存用户 ID，用户资料归统一认证库管理”。SQLite 场景下不强求物理外键跨库约束，业务层负责校验。

## 6. 上传文件分离设计

### 6.1 上传目录

质量检查上传字段统一配置到：

```python
uploadfolder=APP_UPLOAD_DIR
```

涉及字段：

- `inspection_product.drawing_file`
- `inspection_process.drawing_file`
- `inspection_process_page.page_file`
- `inspection_record.site_photo`
- `inspection_record.attachment`
- `inspection_record_photo.photo_file`

在参考开发环境中，该目录配置为：

```text
runtime_data/<web2py_service>/<quality_app>/uploads/
```

在生产环境中，可以映射到：

```text
/data/<web2py_service>/<quality_app>/uploads/
```

关键点：业务代码不再自己拼 `request.folder/uploads`，而是统一读 `01_db.py` 已解析好的 `APP_UPLOAD_DIR`。

### 6.2 上传入口

`controllers/default.py` 中 `_store_upload()` 封装了上传保存：

```python
return field.store(fileobj, filename)
```

好处：

- 复用 web2py upload 字段的命名和保存规则。
- 下载时可以继续使用 `response.download(request, _db)`。
- 文件名由字段管理，业务表只保存文件标识。

### 6.3 多图和软删除

检测记录要求至少上传一张现场照片。主记录保留首图：

```text
inspection_record.site_photo
```

多图保存在：

```text
inspection_record_photo
```

删除照片时不直接删除物理文件，而是标记：

```text
is_deleted = True
deleted_by
deleted_on
```

这有利于现场记录追溯和异常闭环。

### 6.4 下载入口

下载接口：

```python
def download():
    return response.download(request, _db)

def inline_file():
    return response.download(request, _db, attachment=False)
```

必须传入 `_db`，因为上传字段定义在 `db_app`，不在认证库 `db`。

## 7. 权限分离设计

### 7.1 角色存储

质量检查权限角色存储在 `service_center` 的认证库中，实际路径由 `auth_root` 决定：

```text
auth_root/databases/storage.sqlite
  auth_group
  auth_membership
```

`<quality_app>/models/40_permissions.py` 在启动时初始化角色。

### 7.2 角色命名

角色命名规范：

```text
inspection_{范围}_{操作}
```

内置角色：

```text
inspection_super_admin
inspection_admin
inspection_全部_查看
inspection_全部_录入
inspection_全部_审核
inspection_质量部_查看
inspection_质量部_录入
inspection_质量部_审核
inspection_质量部_管理
inspection_生产部_...
inspection_技术部_...
```

### 7.3 权限判断

权限判断函数：

```python
has_inspection_permission(action, scope=None)
```

判断优先级：

1. `inspection_super_admin`
2. `inspection_全部_{操作}`
3. `inspection_{部门}_{操作}`

这样权限数据统一在平台管理后台维护，业务 app 只负责执行权限判断。

## 8. 导航分离设计

`<quality_app>` 不维护独立平台顶部菜单，而是在 `models/30_menu.py` 加载 `service_center/models/xx_menu.py`。

平台菜单源在：

```text
applications/service_center/modules/platform_nav.py
```

其中包含质量检查入口：

```python
dict(
    seed_key='top.<quality_app>',
    title='质量检查',
    url=URL('<quality_app>', 'default', 'dashboard'),
    icon='fas fa-clipboard-check',
    sort_order=25,
    match_app='<quality_app>',
)
```

菜单边界：

- 平台顶部导航由 `service_center` 管。
- `<quality_app>` 内部页面导航、面包屑、页面按钮由 `<quality_app>` 管。

## 9. 配置策略

当前推荐 `<quality_app>/private/appconfig.ini` 同时承担应用配置、会话配置和运行数据路径配置：

```ini
[app]
name = Inspection
production = false
toolbar = false

[db]
uri = sqlite://storage.sqlite
migrate = true

[db_app]
uri = sqlite://db_app.sqlite
migrate = true

[runtime]
root = /data/<web2py_service>/<quality_app>
auth_root = /data/<web2py_service>/service_center

[db]
uri = sqlite://storage.sqlite
migrate = true
pool_size = 10

[session]
masterapp = service_center
```

注意：

- `[db]` 配置会被 `service_center_auth` 读取，影响认证库迁移行为。
- 质量检查业务库 URI、迁移开关和目录已从 `01_db.py` 统一读取。
- 真实 `private/appconfig.ini` 不进入 Git。
- 应补充并提交 `private/appconfig.ini.example`，用于说明部署所需配置。

## 9.1 运行路径诊断页

`<quality_app>` 现已在设置页提供只读运行路径诊断，展示：

- 运行根目录
- 业务数据库目录
- 上传目录
- 认证根目录和认证数据库目录

这一页用于确认“当前到底在读哪里”，特别适合排查：

- 数据迁移后路径未切换

- 生产环境环境变量覆盖了 `appconfig.ini`
- 目录存在但不可写

## 10. 部署目录建议

### 10.1 开发环境

参考开发环境可以使用独立运行目录，并通过配置显式指定：

```text
runtime_data/<web2py_service>/<quality_app>/databases/
runtime_data/<web2py_service>/<quality_app>/uploads/
runtime_data/<web2py_service>/<quality_app>/sessions/
runtime_data/<web2py_service>/<quality_app>/errors/
runtime_data/<web2py_service>/<quality_app>/cache/
```

这些目录由 `.gitignore` 排除，不随应用代码发布。

### 10.2 生产环境

生产环境建议将运行数据放到：

```text
/data/<web2py_service>/<quality_app>/
  databases/
  uploads/
  sessions/
  errors/
  cache/
```

并在 `private/appconfig.ini` 中显式写出每个目录。这样系统迁移时只需要迁移数据目录和配置清单，不需要检查软链接是否断链、是否相对路径失效。

`sessions` 通常由 `service_center` 统一管理，`<quality_app>/sessions` 可以保留为空目录或按 web2py 需要创建。业务数据库和上传目录已由模型直接接管；`sessions/errors/cache` 是否完全迁出，还要结合 web2py 启动层和部署方式一起处理。

### 10.3 启动前检查

启动前应检查：

```bash
test -w /data/<web2py_service>/<quality_app>/databases
test -w /data/<web2py_service>/<quality_app>/uploads
test -w /data/<web2py_service>/<quality_app>/errors
test -w /data/<web2py_service>/service_center/databases
test -w applications/service_center/sessions
```

如果目录是软链接，还应检查是否断链：

```bash
find applications -maxdepth 2 -xtype l -print
```

必须备份：

```text
${SERVICE_CENTER_RUNTIME_ROOT}/databases/storage.sqlite
${<QUALITY_APP>_RUNTIME_ROOT}/databases/db_app.sqlite
${<QUALITY_APP>_RUNTIME_ROOT}/databases/*.table
${<QUALITY_APP>_RUNTIME_ROOT}/uploads/
```

建议短期保留：

```text
<quality_app>/errors/
```

可以清理或重建：

```text
<quality_app>/cache/
<quality_app>/sessions/
```

备份原则：

- 代码包和数据包分开。
- 代码备份不要跟随软链接。
- 数据备份以 `/data/<web2py_service>` 或实际运行数据根目录为准。
- SQLite 备份前最好停服务，或使用 SQLite 在线备份方式。

## 12. 从旧 app 迁移到 <quality_app> 模式

其他 app 可以按以下步骤迁移。

### 12.0 模型文件编号规范（精简版）

web2py 会按模型文件名排序执行 `models/*.py`。因此模型文件确实需要有顺序意识，但不代表必须拆成很多很薄的文件。

结合当前项目的组织习惯，更推荐把编号理解为“内容层级 + 执行顺序”，并给每层预留扩展空间：

- `00`：app 基本信息层。
- `01`：共享模块和数据库层。
- `20`：app 私有业务数据层。
- `30`：菜单导航层。
- `40`：app 私有权限层。
- `90+`：低频任务、兼容逻辑、诊断逻辑。

也就是说，`00` 和 `01` 是启动基础，必须最早执行；`20` 开始进入本 app 的业务数据定义；`30` 和 `40` 再处理依赖前面对象的菜单、权限和初始化扩展。

本项目更推荐的原则是：

- 先保证执行顺序清楚。
- 再尽量减少文件数量。
- 只在存在明显初始化边界时拆文件。
- 优先按“共享层 / 业务层 / 扩展层”来拆，而不是按细碎职责拆。

换句话说，编号的目的是表达依赖，不是把每个职责都强行拆成一个文件。

#### 推荐拆分尺度

对于大多数业务 app，`models/` 保持 **2 到 4 个文件** 就够了。

推荐结构：

| 编号 | 文件示例 | 职责 |
| --- | --- | --- |
| `00` | `00_app.py` | app 名称、页面标题、资源版本、基础响应元数据 |
| `01` | `01_db.py` | 共享模块接入、统一认证、认证库对象、业务库连接 |
| `20` | `20_tables.py` | app 自身业务常量、`define_table()`、字段校验 |
| `30` | `30_menu.py` | 平台统一菜单接入、app 菜单兜底逻辑 |
| `40` | `40_permissions.py` | app 权限、角色初始化、访问控制补充逻辑 |
| `90` | `90_tasks.py` | 可选，定时任务或后台任务注册 |

如果应用更小，甚至可以进一步合并为：

| 编号 | 文件示例 | 职责 |
| --- | --- | --- |
| `00` | `00_app.py` | app 基本信息 |
| `01` | `01_bootstrap.py` | 共享模块接入、数据库连接、业务表、菜单、权限 |

#### 何时应该拆文件

只有下面几种情况，拆文件才是值得的：

- 后续文件必须依赖前面的初始化结果。
- 表定义本身已经很长，继续和认证、菜单混在一起会难读。
- 某块逻辑会被多个 app 共享，例如统一认证、统一菜单、统一运行路径。
- 某块逻辑有明显启动顺序要求，例如数据库连接必须早于业务表定义。

#### 何时不应该再拆

如果某个模型文件只有几十行，而且与前后文件强耦合，就不值得单独存在。例如：

- 单独一个 `30_menu.py`，只有加载统一菜单的十几行。
- 单独一个 `40_permissions.py`，但权限逻辑只服务本 app 且规模不大。
- 单独一个 `00_app.py`，只放几个标题变量，但未来又想再建 `00_runtime.py`。

这种拆法会让人需要在多个文件间来回跳，读起来比一个稍长但顺序清楚的文件更累。

#### 对 <quality_app> 的建议

当前：

```text
00_app.py
01_db.py
20_tables.py
30_menu.py
40_permissions.py
```

这个命名已经比较合理，属于“低风险语义化拆分”。当前不需要继续合并成更少文件。

每层含义：

- `00_app.py`：app 基本信息，不放数据库和业务表。
- `01_db.py`：共享模块和数据库初始化，负责产出 `db`、`auth`、`db_app`。
- `20_tables.py`：私有业务数据表，依赖 `01_db.py`。
- `30_menu.py`：菜单导航，依赖平台菜单模块。
- `40_permissions.py`：私有权限和角色初始化，依赖认证库已初始化。

#### 最重要的约束

无论拆成几个文件，都要满足下面顺序：

1. `00_app.py` 先初始化 app 基本信息。
2. `01_db.py` 完成共享模块接入、统一认证和数据库连接。
3. `20_tables.py` 定义业务表和字段校验。
4. `30_menu.py` 设置菜单。
5. `40_permissions.py` 最后补充权限和幂等角色初始化。

所以真正应该追求的是“少文件但顺序稳定”，而不是“职责拆得越细越好”。

在目标 app 的 `models/00_db.py` 或 `models/01_db.py`：


### 12.3 迁移业务表

将：

```python
db.define_table("business_table", ...)
```

改为：

```python

```

认证用户校验继续使用：


```python
IS_IN_DB(_app_db, "business_table.id", "%(name)s")
```

## 13. 风险和约束

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| `db` 和 `_app_db` 混用 | 业务表写入认证库或查询失败 | 业务表统一使用 `_app_db` |
| 下载接口传错数据库 | 上传文件无法下载 | `response.download(request, _db)` |
| 上传目录不可写 | 现场照片和指导书上传失败 | 启动前检查 `uploads` 权限 |
| 认证库丢失 | 所有平台用户无法登录 | 备份 `service_center/databases` |
| 业务库丢失 | 质量检查数据丢失 | 备份 `<quality_app>/databases` |
| 只备份代码不备份数据 | 恢复后业务空库 | 代码包、数据包分开备份 |
| 软链接断链 | web2py 写库或上传时报错 | 部署脚本检查 `-xtype l` |

## 14. 验收清单

基础验收：

- 访问 `/<quality_app>/default/dashboard`，未登录时跳转统一登录。
- 使用 `service_center` 用户登录后可进入质量检查首页。
- 顶部导航出现“质量检查”入口。
- `inspection_admin` 或质量检查角色可访问管理功能。

数据库验收：

- 当前认证运行目录的 `databases/storage.sqlite` 中存在 `auth_user`、`auth_group`、`auth_membership`。
- 当前质量检查运行目录的 `databases/db_app.sqlite` 中存在 `inspection_product`、`inspection_process`、`inspection_checkpoint`、`inspection_record`。
- 新增质量检查业务数据后，只写入 `db_app.sqlite`。
- 新增角色和用户授权后，只写入 `service_center` 认证库。

上传验收：

- 质量作业指导书文件可上传。
- 检查点图样页可上传。
- 每条检测记录至少一张现场照片的校验生效。
- 检测记录详情页可追加照片。
- 已上传文件可通过 `download` 或 `inline_file` 正常访问。

运维验收：

- `databases`、`uploads`、`sessions`、`errors`、`cache` 不进入 Git。
- 清理 `cache` 后页面可重新生成。
- 清理 `sessions` 后用户重新登录即可恢复。
- 备份包同时包含认证库、质量检查业务库和上传目录。

## 15. 推荐后续增强

1. 为更多 app 复制 `<quality_app>` 的 `01_db.py + 诊断页` 模式。
2. 增加启动前目录诊断脚本，统一检查 `service_center` 和业务 app 数据目录。
3. 生产环境将 `<quality_app>/databases`、`<quality_app>/uploads` 映射到 `/data/<web2py_service>/<quality_app>`。
4. 默认管理员密码不要固化在代码中，改为部署初始化命令或首次启动环境变量。
5. 为关键业务操作增加审计日志，记录新增、修改、删除、异常关闭等动作。

## 16. 一句话总结

`<quality_app>` 的最新分离方案是：统一平台能力归 `service_center`，质量检查业务能力归 `<quality_app>`；认证库和业务库分开；上传文件跟业务库字段绑定；运行目录使用 web2py 标准目录名，由部署层决定真实数据位置。
