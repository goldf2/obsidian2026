# <quality_app> 生产数据分离方案

> 归档说明: 本文已并入 `数据代码分离/数据代码分离统一规范.md`，只作为历史参考保留，不再作为当前执行依据。

本文定义 `<quality_app>` 在生产环境中的 web2py 数据目录重定位方式。目标是先看清 web2py 默认会在 app 目录下产生哪些运行数据，再逐一决定哪些目录必须重定位、哪些目录保留默认行为、哪些目录后续再治理。

当前执行口径已改为 `数据代码分离/数据代码分离统一规范.md`。本文中的实现示例仅用于追溯旧方案如何收敛到 `storage.upload_root` + `UploadField` 封装。

本方案不使用软链接。生产环境通过配置和代码显式指定数据目录，避免系统迁移时出现断链、复制语义不一致或隐藏路径依赖。

## 1. web2py 默认数据目录

web2py 的每个 app 通常位于：

```text
/opt/<web2py_service>/applications/<quality_app>
```

在默认模式下，以下目录可能包含运行时数据。

| 默认目录 | 用途 | 是否属于业务数据 | 生产处理建议 |
| --- | --- | --- | --- |
| `databases/` | SQLite 数据库文件、DAL 表结构元数据、迁移记录、`sql.log` 等 | 是 | 必须重定位 |
| `uploads/` | `Field(..., "upload")` 上传附件默认存放位置 | 是 | 必须重定位 |
| `sessions/` | 文件型 session 数据 | 部分是 | 第二阶段处理 |
| `errors/` | web2py ticket 错误文件 | 运维数据 | 第二阶段处理 |
| `cache/` | web2py 缓存文件 | 临时数据 | 可保留或第二阶段处理 |
| `private/` | 私有配置、密钥、只读模板、运行配置 | 配置数据 | 生产配置外置 |
| `cron/` | app 级定时任务脚本 | 代码/任务定义 | 保留在代码目录 |
| `static/` | CSS、JS、图片等静态资源 | 代码资源 | 保留在代码目录 |
| `views/` | 页面模板 | 代码资源 | 保留在代码目录 |
| `models/` | 模型代码和表定义 | 代码资源 | 保留在代码目录 |
| `controllers/` | 控制器代码 | 代码资源 | 保留在代码目录 |
| `modules/` | app 私有 Python 模块 | 代码资源 | 保留在代码目录 |
| `languages/` | 翻译文件，开发时可能被 web2py 写入 | 代码/构建产物 | 生产只读或受控更新 |

数据分离的核心不是把所有目录都移动到 `/data`，而是明确分类：

- 业务持久数据：数据库、上传附件、业务导出、备份。
- 框架运行数据：sessions、errors、cache。
- 代码和资源：models、controllers、modules、views、static、cron、languages。
- 配置数据：生产配置文件和密钥。

## 2. 生产目标结构

生产环境固定分为两类目录：

```text
/opt/<web2py_service>/applications/<quality_app>    # 应用代码
/data/<web2py_service>/<quality_app>                # <quality_app> 运行数据
```

推荐的数据目录结构：

```text
/data/<web2py_service>/<quality_app>/
├── config/
│   └── appconfig.ini
├── databases/
│   └── db_app.sqlite
├── uploads/
│   ├── photos/
│   ├── drawings/
│   └── attachments/
├── backups/
└── runtime/
    ├── cache/
    ├── errors/
    ├── logs/
    └── sessions/
```

生产原则：

- 业务数据库、上传附件、备份、导出文件必须放在 `/data/<web2py_service>/<quality_app>`。
- `/opt/<web2py_service>/applications/<quality_app>` 只存放代码、模板、静态资源和默认配置模板。
- 不把 `/opt/.../databases`、`/opt/.../uploads` 作为生产数据目录。
- 不使用软链接把 app 内目录指向 `/data`。

## 3. 逐目录解决方案

### 3.1 `databases/`

用途：

- web2py 默认把 SQLite 数据库文件放在 app 的 `databases/`。
- DAL 迁移表文件、表结构元数据和 `sql.log` 也通常在这里。

生产方案：

- <quality_app> 业务库必须重定位到 `/data/<web2py_service>/<quality_app>/databases`。
- 使用 DAL 的 `folder` 参数显式指定目录。
- 数据库文件名仍保持简单，例如 `sqlite://db_app.sqlite`。

示例：

```python
db_app = DAL(
    "sqlite://db_app.sqlite",
    folder="/data/<web2py_service>/<quality_app>/databases",
    migrate=True,
    pool_size=10,
)
```

要求：

- 不依赖 app 内 `databases/`。
- 不使用软链接。
- 迁移时复制旧数据库文件和 DAL 表结构文件到 `/data/.../databases/`。

### 3.2 `uploads/`

用途：

- web2py 的 `Field("x", "upload")` 默认使用 app 内 `uploads/`。
- 现场照片、图纸、PDF、附件等都属于业务持久数据。

生产方案：

- 上传目录必须重定位到 `/data/<web2py_service>/<quality_app>/uploads`。
- 按业务类型拆分子目录。
- upload 字段统一使用模型层封装，由封装内部设置 `uploadfolder`。

示例：

```python
<QUALITY_APP>_UPLOAD_ROOT = "/data/<web2py_service>/<quality_app>/uploads"

UploadField("attachment", label=T("附件"))
UploadField("site_photo", subdir="photos", label=T("现场照片"))
UploadField("drawing_file", subdir="drawings", label=T("图样文件"))
```

要求：

- 表定义只使用模型层暴露的 `UploadField` / `<QUALITY_APP>_UPLOAD_FIELD`。
- 控制器不直接拼接上传目录。
- 下载继续使用 `response.download(request, db_app)`，确保 DAL 知道 upload 字段对应的存储规则。

### 3.3 `private/`

用途：

- 默认存放 `appconfig.ini`、密钥、内部配置和不应公开访问的文件。
- 在生产环境中，真实配置和密钥也属于需要独立备份和保护的数据。

生产方案：

- app 内保留 `private/appconfig.ini.example` 作为模板。
- 真实生产配置放到 `/data/<web2py_service>/<quality_app>/config/appconfig.ini`。
- 代码优先读取 `/data/.../config/appconfig.ini`，找不到时再回退到 app 内配置或默认值。

示例配置：

```ini
[storage]
runtime_root = /data/<web2py_service>/<quality_app>

[db_app]
uri = sqlite://db_app.sqlite
migrate = true
pool_size = 10
```

要求：

- 生产配置不提交到 Git。
- 不用软链接把 `private/appconfig.ini` 指向 `/data/.../config/appconfig.ini`。
- 配置入口尽量保持一个核心值：`runtime_root`。

### 3.4 `sessions/`

用途：

- web2py 默认文件型 session 会写入 app 内 `sessions/`。
- <quality_app> 当前使用统一认证，session 还会受 `service_center` 的 `masterapp` 策略影响。

生产方案：

- 第一阶段不强制重定位 <quality_app> 的 session 目录。
- 优先使用统一认证的 masterapp session 或后续切换为数据库 session。
- 如果必须管理文件 session，应通过 web2py 支持的 session 配置或启动层策略处理，不使用软链接。

要求：

- 不把 session 和业务数据混为一类。
- session 迁移不是业务数据迁移的前置条件。
- session 可以通过过期策略或重登录解决。

### 3.5 `errors/`

用途：

- web2py ticket 错误文件默认写入 app 内 `errors/`。
- 它是故障诊断资料，不是业务主数据。

生产方案：

- 第一阶段可以保留默认目录。
- 运维上定期归档或清理。
- 后续可以接入日志采集、统一 ticket 收集或错误目录外置。

要求：

- 不因为 errors 目录未重定位而阻塞业务数据分离。
- 迁移时可选择保留近期 ticket，但不作为核心业务数据。

### 3.6 `cache/`

用途：

- web2py 或 app 可能使用文件缓存。
- 缓存属于可再生临时数据。

生产方案：

- 第一阶段可以保留默认目录或在启动时清理。
- 不纳入业务备份。
- 如后续性能需要，再统一迁移到 `/data/<web2py_service>/<quality_app>/runtime/cache` 或 Redis 等外部缓存。

要求：

- 不用软链接处理。
- 不把 cache 纳入数据迁移的必要步骤。

### 3.7 `languages/`

用途：

- web2py 翻译文件目录。
- 开发模式下 web2py 可能尝试更新翻译文件。

生产方案：

- 作为代码资源管理。
- 生产环境尽量只读。
- 需要更新翻译时走代码发布流程，而不是运行时任意写入。

要求：

- 不纳入 `/data` 运行数据。
- 如果日志中出现无法写入语言文件的 warning，优先判断是否影响业务功能。

### 3.8 `models/`、`controllers/`、`modules/`、`views/`、`static/`、`cron/`

用途：

- app 代码、模板、静态资源和定时任务定义。

生产方案：

- 全部保留在 `/opt/<web2py_service>/applications/<quality_app>`。
- 通过代码发布和版本管理维护。
- 不迁移到 `/data`。

要求：

- 运行时不要在这些目录生成业务数据。
- cron 如果会生成导出文件或备份文件，输出目录必须显式写到 `/data/<web2py_service>/<quality_app>/backups` 或其他 `/data` 子目录。

## 4. 统一配置实现

生产环境只需要一个核心配置：`runtime_root`。

推荐配置文件位置：

```text
/data/<web2py_service>/<quality_app>/config/appconfig.ini
```

最小配置：

```ini
[storage]
runtime_root = /data/<web2py_service>/<quality_app>

[db_app]
uri = sqlite://db_app.sqlite
migrate = true
pool_size = 10
```

路径解析集中在一个模块中，例如 `modules/app_config.py`：

```python
import os
from gluon.contrib.appconfig import AppConfig

APP_NAME = "<quality_app>"
DEFAULT_DATA_ROOT = os.environ.get("WEB2PY_DATA_ROOT", "/data/<web2py_service>")
DEFAULT_RUNTIME_ROOT = os.path.join(DEFAULT_DATA_ROOT, APP_NAME)


class InspectionRuntimeConfig(object):
    def __init__(self, request):
        self.request = request
        self.app_folder = request.folder
        self.configuration = self._load_configuration()

        self.runtime_root = self._config_value(
            "storage.runtime_root",
            DEFAULT_RUNTIME_ROOT,
        )
        self.database_folder = self._storage_path("storage.database_folder", "databases")
        self.upload_root = self._storage_path("storage.upload_root", "uploads")
        self.photo_folder = self._storage_path("storage.photo_folder", "uploads/photos")
        self.drawing_folder = self._storage_path("storage.drawing_folder", "uploads/drawings")
        self.attachment_folder = self._storage_path(
            "storage.attachment_folder",
            "uploads/attachments",
        )
        self.backup_folder = self._storage_path("storage.backup_folder", "backups")

        self.db_uri = self._config_value(
            "db_app.uri",
            "sqlite://db_app.sqlite",
        )
        self.db_migrate = self._as_bool(self._config_value("db_app.migrate", "true"), True)
        self.db_pool_size = int(self._config_value("db_app.pool_size", 10))

    def _load_configuration(self):
        external = os.path.join(DEFAULT_RUNTIME_ROOT, "config", "appconfig.ini")
        if os.path.exists(external):
            return AppConfig(external, reload=True)
        return AppConfig(reload=True)

    def _config_value(self, key, default=None):
        value = self.configuration.get(key)
        return str(value).strip() if value not in (None, "") else default

    def _storage_path(self, key, relative_default):
        value = self._config_value(key)
        if not value:
            value = os.path.join(self.runtime_root, relative_default)
        if not os.path.isabs(value):
            value = os.path.join(self.runtime_root, value)
        return os.path.abspath(value)

    def _as_bool(self, value, default=False):
        if value in (None, ""):
            return default
        return str(value).strip().lower() in ("1", "true", "yes", "on")
```

`models/01_db.py` 只负责加载配置、创建数据库连接并导出统一上传字段封装：

```python
runtime_config = InspectionRuntimeConfig(request)

db_app = DAL(
    runtime_config.db_uri,
    migrate=runtime_config.db_migrate,
    pool_size=runtime_config.db_pool_size,
    folder=runtime_config.database_folder,
)

<QUALITY_APP>_RUNTIME_CONFIG = runtime_config
<QUALITY_APP>_UPLOAD_ROOT = runtime_config.upload_root
<QUALITY_APP>_UPLOAD_PATH = runtime_config.upload_path


def <QUALITY_APP>_UPLOAD_FIELD(name, *args, **kwargs):
    subdir = kwargs.pop("subdir", None)
    kwargs.setdefault(
        "uploadfolder",
        <QUALITY_APP>_UPLOAD_PATH(subdir) if subdir else <QUALITY_APP>_UPLOAD_ROOT,
    )
    return Field(name, "upload", *args, **kwargs)


UploadField = <QUALITY_APP>_UPLOAD_FIELD
```

表定义只消费模型暴露的上传字段封装：

```python
UploadField("attachment", label=T("附件"))
UploadField("site_photo", subdir="photos", label=T("现场照片"))
UploadField("drawing_file", subdir="drawings", label=T("图样文件"))
```

核心规则：

- DAL 的 `folder` 决定 SQLite 数据库文件位置。
- `UploadField` 内部设置的 `uploadfolder` 决定附件文件位置。
- 控制器不拼接数据目录。
- 表定义不直接写 `/data/...` 字符串。

## 5. 部署初始化

生产部署时先创建目录并设置权限：

```bash
install -d -o nobody -g nogroup /data/<web2py_service>/<quality_app>/config
install -d -o nobody -g nogroup /data/<web2py_service>/<quality_app>/databases
install -d -o nobody -g nogroup /data/<web2py_service>/<quality_app>/uploads/photos
install -d -o nobody -g nogroup /data/<web2py_service>/<quality_app>/uploads/drawings
install -d -o nobody -g nogroup /data/<web2py_service>/<quality_app>/uploads/attachments
install -d -o nobody -g nogroup /data/<web2py_service>/<quality_app>/backups
install -d -o nobody -g nogroup /data/<web2py_service>/<quality_app>/runtime/cache
install -d -o nobody -g nogroup /data/<web2py_service>/<quality_app>/runtime/errors
install -d -o nobody -g nogroup /data/<web2py_service>/<quality_app>/runtime/logs
install -d -o nobody -g nogroup /data/<web2py_service>/<quality_app>/runtime/sessions
```

然后放置生产配置：

```text
/data/<web2py_service>/<quality_app>/config/appconfig.ini
```

注意：

- 目录创建建议由部署脚本或运维命令完成。
- 应用启动时可以检查目录是否存在和可写，但不要依赖请求过程大量创建生产目录。
- 配置文件不要提交到 Git，模板文件可以提交为 `private/appconfig.ini.example`。

## 6. 迁移流程

从旧目录迁移到 `/data` 时按以下顺序执行：

1. 停止 web2py 服务。
2. 备份旧 app 目录中的 `databases` 和 `uploads`。
3. 创建 `/data/<web2py_service>/<quality_app>` 目录结构。
4. 复制旧业务数据库到 `/data/<web2py_service>/<quality_app>/databases/`。
5. 复制旧上传文件到 `/data/<web2py_service>/<quality_app>/uploads/`。
6. 写入 `/data/<web2py_service>/<quality_app>/config/appconfig.ini`。
7. 检查目录属主为 web2py 运行用户。
8. 启动 web2py 服务。
9. 验证登录、录入、上传、下载、历史附件访问。

建议迁移命令使用显式源目录和目标目录，不使用软链接：

```bash
rsync -a /opt/<web2py_service>/applications/<quality_app>/databases/ \
  /data/<web2py_service>/<quality_app>/databases/

rsync -a /opt/<web2py_service>/applications/<quality_app>/uploads/ \
  /data/<web2py_service>/<quality_app>/uploads/
```

迁移后不要删除旧目录，先改名保留观察期：

```bash
mv /opt/<web2py_service>/applications/<quality_app>/databases \
  /opt/<web2py_service>/applications/<quality_app>/databases.disabled

mv /opt/<web2py_service>/applications/<quality_app>/uploads \
  /opt/<web2py_service>/applications/<quality_app>/uploads.disabled
```

观察期确认无问题后再清理。

## 7. 验收检查

必须通过以下检查：

```bash
test -f /data/<web2py_service>/<quality_app>/databases/db_app.sqlite
test -w /data/<web2py_service>/<quality_app>/databases
test -w /data/<web2py_service>/<quality_app>/uploads/photos
test -w /data/<web2py_service>/<quality_app>/uploads/drawings
test -w /data/<web2py_service>/<quality_app>/uploads/attachments
```

应用侧验收：

- 新建质量检查记录后，业务数据写入 `/data/<web2py_service>/<quality_app>/databases/db_app.sqlite`。
- 上传现场照片后，文件出现在 `/data/<web2py_service>/<quality_app>/uploads/photos`。
- 上传图纸后，文件出现在 `/data/<web2py_service>/<quality_app>/uploads/drawings`。
- 历史附件可以通过 `response.download(request, db_app)` 下载。
- app 代码目录不再产生新的业务数据库和上传附件。

## 8. 诊断页建议

可以增加只读诊断页，展示：

- 当前配置文件路径。
- `runtime_root`。
- 数据库目录、上传目录、备份目录。
- 目录是否存在。
- 目录是否可写。
- 目录文件数量。
- 最近上传文件时间。
- app 内默认 `databases/` 和 `uploads/` 是否仍有新文件产生。

诊断页只读，不直接修改数据库目录。路径修改应通过配置文件和服务重启完成。

## 9. 总结

可靠的数据分离不依赖软链接，而依赖显式配置：

- 先识别 web2py 默认运行数据目录。
- 业务数据库用 DAL `folder` 重定位。
- 上传附件用字段级 `uploadfolder` 重定位。
- 真实生产配置放到 `/data/<web2py_service>/<quality_app>/config/appconfig.ini`。
- sessions、errors、cache 不作为第一阶段业务数据迁移目标，后续按框架运行数据单独治理。
- 代码、模板、静态资源、模块和 cron 定义保留在 `/opt/<web2py_service>/applications/<quality_app>`。

这样迁移时只需要迁移两块内容：

```text
/opt/<web2py_service>    # 代码
/data/<web2py_service>   # 数据
```

目录关系清楚，恢复过程可重复，也不会因为软链接断裂导致服务异常。
