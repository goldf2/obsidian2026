# <web2py_service> 运行数据目录配置化升级方案

版本：v1.0  
日期：2026-06-05  
适用范围：`/opt/<web2py_service>` 管理平台及其 web2py 应用集群

## 1. 背景

当前平台为了实现代码和运行数据分离，较多使用软连接，将各 app 下的 `databases`、`sessions`、`errors`、`uploads`、`cache` 等目录指向统一运行数据目录。

这种方式可以快速解决代码目录和数据目录混放的问题，但也带来了几个维护风险：

- 软连接路径依赖本机目录，路径变化后容易失效。
- 软连接失效时，web2py 报错不够直观，容易出现 ticket 不一致或 `unknown`。
- 多 app 软连接数量较多，人工维护成本高。
- 部署、迁移、备份时不容易一眼确认哪些是代码，哪些是运行数据。
- Git 状态容易被运行数据目录、错误票据、会话文件干扰。

本方案目标是逐步从“软连接数据分离”升级为“配置驱动运行目录”，最终尽量减少或取消软连接。

## 2. 数据分级原则

运行目录应按数据价值分级处理，不需要对所有目录采用同等保护策略。

### 2.1 必须妥善保存

这些目录和文件属于业务数据或关键配置，迁移、升级、备份时必须重点保护：

- `databases`：SQLite 数据库、迁移元数据、业务主数据。
- `uploads`：附件、现场照片、指导书文件、图片和 PDF。
- 关键配置文件：平台启动配置、数据库连接配置、必要密钥配置。

### 2.2 可以舍弃或定期清理

这些目录属于运行临时数据，可以在停机维护、升级、迁移时清理或重建：

- `sessions`：登录会话，清理后用户重新登录即可。
- `errors`：错误 ticket，适合短期排错，不作为长期业务数据。
- `cache`：缓存，可重建。
- `logs`：普通访问日志可轮转归档，非核心业务数据。

### 2.3 方案影响

基于以上分级，本方案不再追求第一阶段完整迁移 `sessions`、`errors`、`cache`。升级重点调整为：

- 优先保护和配置化 `databases`。
- 优先保护和配置化 `uploads`。
- 临时目录由启动层创建和清理即可。
- 错误 ticket 只保留短期排查价值，不作为核心备份对象。

## 3. 升级目标

### 3.1 主要目标

- 使用统一配置文件声明各 app 的运行数据目录。
- 启动时由执行层读取配置、创建目录、校验权限。
- app 内部通过运行路径解析模块获取数据库、上传等核心数据目录。
- 逐步减少软连接，优先将数据库和上传目录改为配置读取。
- 保留回滚能力，避免一次性改动导致平台不可用。

### 3.2 非目标

- 不在第一阶段重构全部 web2py 框架行为。
- 不在第一阶段强行取消所有软连接。
- 不改变现有业务表结构。
- 不改变现有 app 的 URL 入口和用户登录体系。
- 不把 `sessions`、`errors`、`cache` 当作核心业务数据长期保存。

## 4. 推荐总体结构

推荐将运行数据目录放到项目外部的 `/data/<web2py_service>`，并由配置统一声明。

代码目录：

```text
/opt/<web2py_service>/
  runtime.json
  scripts/
    run_platform.py
    prepare_runtime_dirs.py
  applications/
    service_center/
    <quality_app>/
    <business_app>/
```

运行数据目录：

```text
/data/<web2py_service>/
  service_center/
  <quality_app>/
  <business_app>/
```

其中：

- `runtime.json`：声明运行数据根目录和各 app 数据目录。
- `scripts/prepare_runtime_dirs.py`：创建目录、检查权限、输出诊断。
- `scripts/run_platform.py`：启动执行层，读取配置后启动 web2py。
- `applications/*/models/00_runtime.py`：app 级运行路径初始化。

## 5. 配置文件设计

建议先使用 JSON，不使用 YAML。

原因：

- JSON 可由 Python 标准库直接读取。
- 不依赖 PyYAML，降低部署复杂度。
- 格式足够表达当前运行目录需求。

建议配置文件路径：

```text
/opt/<web2py_service>/runtime.json
```

示例：

```json
{
  "runtime_base": "/data/<web2py_service>",
  "apps": {
    "service_center": {
      "databases": "service_center/databases",
      "sessions": "service_center/sessions",
      "errors": "service_center/errors",
      "uploads": "service_center/uploads",
      "cache": "service_center/cache"
    },
    "<quality_app>": {
      "databases": "<quality_app>/databases",
      "sessions": "<quality_app>/sessions",
      "errors": "<quality_app>/errors",
      "uploads": "<quality_app>/uploads",
      "cache": "<quality_app>/cache"
    },
    "<business_app>": {
      "databases": "<business_app>/databases",
      "sessions": "<business_app>/sessions",
      "errors": "<business_app>/errors",
      "uploads": "<business_app>/uploads",
      "cache": "<business_app>/cache"
    }
  }
}
```

路径解析规则：

- `runtime_base` 使用绝对路径。
- app 内部目录使用相对路径。
- 最终目录由 `runtime_base + app 配置相对路径` 拼接。
- 禁止配置指向项目代码目录下的 `controllers`、`models`、`views`、`modules`、`static`。

## 6. 执行层设计

配置文件本身不会改变 web2py 的默认行为，所以需要一个执行层。

执行层建议分为两个脚本：

```text
scripts/prepare_runtime_dirs.py
scripts/run_platform.py
```

### 6.1 prepare_runtime_dirs.py

职责：

- 读取 `runtime.json`。
- 解析各 app 的运行数据目录。
- 创建缺失目录。
- 校验目录是否可读写。
- 检查是否存在错误软连接。
- 输出诊断报告。

建议输出示例：

```text
[OK] <quality_app>.databases -> /data/<web2py_service>/<quality_app>/databases
[OK] <quality_app>.uploads   -> /data/<web2py_service>/<quality_app>/uploads
[WARN] <quality_app>.errors currently uses symlink
[FAIL] <business_app>.sessions not writable
```

### 6.2 run_platform.py

职责：

- 执行 `prepare_runtime_dirs.py` 的核心检查。
- 设置环境变量，例如 `WEB2PY_RUNTIME_CONFIG`。
- 启动 web2py。
- 统一管理端口、日志、pid 文件。

建议启动方式：

```bash
python3 scripts/run_platform.py --port 8000 --password test123
```

执行层启动 web2py 时应避免硬编码目录，并保留日志：

```text
httpserver_8000.log
httpserver_8000.pid
httpserver_8000.out
```

## 7. app 内部运行路径模块

建议新增共享模块：

```text
applications/service_center/modules/runtime_paths.py
```

其他 app 可以通过共享模块读取路径。

核心能力：

- `get_runtime_base()`
- `get_app_runtime(app_name)`
- `get_app_dir(app_name, key)`
- `ensure_app_runtime(app_name)`
- `diagnose_runtime(app_name)`

示例调用：

```python
runtime = get_app_runtime("<quality_app>")
db_folder = runtime["databases"]
uploads_folder = runtime["uploads"]
```

## 8. 可配置目录迁移顺序

不同目录的改造难度不同，建议分阶段处理。

### 8.1 第一优先级：数据库目录

数据库目录最适合先配置化。

web2py DAL 支持通过 `folder` 指定数据库目录。

目标：

```python
db_app = DAL(
    "sqlite://db_app.sqlite",
    folder=runtime_dirs["databases"],
    migrate=True,
)
```

收益：

- 数据库立即脱离 app 目录。
- 备份范围清晰。
- 风险较低。

### 8.2 第二优先级：上传目录

上传目录也适合配置化，但需要检查每个 upload 字段。

目标：

```python
Field(
    "photo_file",
    "upload",
    uploadfolder=runtime_dirs["uploads"],
)
```

同时需要确认：

- 下载接口能从配置目录读取文件。
- 缩略图或附件预览不再依赖默认 `applications/<app>/uploads`。
- 历史上传文件迁移到新目录。

### 8.3 第三优先级：临时目录自动创建

`sessions`、`errors`、`cache` 不作为核心业务数据保存。

升级后建议由执行层统一处理：

- 启动前自动创建目录。
- 可以清空旧会话。
- 可以清理旧缓存。
- 可以按天或按数量轮转错误 ticket。

这些目录不需要复杂迁移，重点是保证服务启动时目录存在且可写。

### 8.4 可选项：数据库 session

如果后续希望减少文件 session 对目录的依赖，可以评估数据库 session。

但这不是当前升级的关键目标。

判断原则：

- 需要集中登录状态时，再考虑数据库 session。
- 单机部署时，文件 session 足够。
- session 数据不纳入业务备份。

### 8.5 可选项：错误 ticket 归档

`errors` 目录不是核心数据，但短期排错有价值。

建议：

- 保留最近 7 到 30 天。
- 重大问题处理完成后归档关键 ticket。
- 平台升级时可以清理历史 ticket。

## 9. 分阶段实施计划

### 阶段 0：基线确认

目标：确认当前服务可稳定运行。

任务：

- 记录当前软连接清单。
- 记录当前数据库、上传、错误、session 文件位置。
- 标记核心数据目录：`databases`、`uploads`。
- 标记临时目录：`sessions`、`errors`、`cache`。
- 确认 `service_center` 和 `<quality_app>` 可登录。
- 确认质量检查关键页面返回 200。
- 建立回滚点。

验收：

- `http://127.0.0.1:8000/service_center` 返回 200。
- `http://127.0.0.1:8000/<quality_app>/dashboard` 登录后返回 200。
- 没有新增错误 ticket。

### 阶段 1：引入配置和执行层

目标：先不改业务逻辑，只建立配置化基础。

任务：

- 新增 `runtime.json`。
- 新增 `scripts/prepare_runtime_dirs.py`。
- 新增 `scripts/run_platform.py`。
- 输出目录诊断报告。
- 保留现有软连接。

验收：

- 执行层能正确识别所有 app 运行目录。
- 缺失目录可以自动创建。
- 错误目录权限能被发现。
- 不影响现有服务启动。

### 阶段 2：数据库目录配置化

目标：将核心数据库目录从软连接转为配置读取。

任务：

- 新增 `runtime_paths.py`。
- 修改 `service_center` 认证库初始化。
- 修改 `<quality_app>` 业务库初始化。
- 逐个检查其他 app 的 DAL 初始化。
- 保留旧软连接作为回滚路径。

验收：

- 数据库从配置目录读取。
- 登录成功。
- 质量检查数据、质量作业指导书、检查点可读取。
- 数据库文件不再依赖 app 下 `databases` 软连接。

### 阶段 3：上传目录配置化

目标：将业务上传文件从默认 uploads 目录迁移到配置目录。

任务：

- 检查所有 `Field(..., "upload")`。
- 为 upload 字段指定 `uploadfolder`。
- 调整下载和预览接口。
- 迁移历史上传文件。
- 做现场照片上传和下载测试。

验收：

- 质量检查现场照片可上传。
- 多图上传可用。
- 图片预览可用。
- 下载接口可用。
- 旧文件不丢失。

### 阶段 4：临时目录策略调整

目标：让临时目录可自动创建、可清理，不再作为重点迁移对象。

任务：

- 启动层自动创建 `sessions`、`errors`、`cache`。
- 启动层检查临时目录是否可写。
- 增加清理策略说明。
- 保留最近一段时间错误 ticket 供排查。
- 明确这些目录不纳入核心备份。

验收：

- 登录状态稳定。
- 错误 ticket 可查看。
- 清理 session 后可重新登录。
- 清理 cache 后页面可正常重新生成。

### 阶段 5：清理软连接

目标：只保留确实无法配置化的少数目录。

任务：

- 删除已经不再需要的软连接。
- 更新部署文档。
- 更新备份策略。
- 更新 Git 忽略规则。
- 建立定期诊断脚本。
- 对临时目录允许重建，不做复杂迁移。

验收：

- 代码目录和数据目录清晰分离。
- 新机器部署可由配置和脚本完成。
- Git 状态不再被运行数据污染。

## 10. 风险和应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 配置路径错误 | 服务无法启动或找不到数据库 | 启动前强制校验目录 |
| 数据库迁移失败 | 登录或业务数据不可用 | 保留原目录和回滚脚本 |
| 上传路径改造遗漏 | 图片无法上传或下载 | 逐表检查 upload 字段 |
| 误删上传文件 | 指导书或现场照片丢失 | 上传目录迁移前完整备份 |
| 临时目录被清理 | 用户需重新登录或 ticket 消失 | 明确临时目录可舍弃策略 |
| 多 app 配置不一致 | 维护成本上升 | 使用统一 runtime_paths 模块 |

## 11. 回滚方案

每个阶段必须可回滚。

### 数据库目录回滚

- 停止服务。
- 将 DAL `folder` 改回原路径。
- 恢复旧数据库目录或软连接。
- 启动服务。
- 验证登录和关键页面。

### 上传目录回滚

- 停止服务。
- 将 upload 字段 `uploadfolder` 改回默认。
- 将迁移文件复制回原 uploads。
- 启动服务。
- 验证上传、预览、下载。

### 执行层回滚

- 停止通过 `scripts/run_platform.py` 启动。
- 使用原 web2py 启动命令。
- 保留 `runtime.json` 不参与运行。

## 12. 验收清单

基础验收：

- 根路径跳转到 `service_center`。
- 统一登录可用。
- `inspection_admin` 可登录。
- 质量检查看板可打开。
- 质量作业指导书列表可打开。
- 质量作业指导书维护页可打开。
- 质量检查采集页可打开。
- 检测记录页可打开。
- 统计报表页可打开。
- 异常闭环页可打开。

数据验收：

- 业务数据库目录来自配置。
- 上传目录来自配置。
- 新上传文件落到配置目录。
- 历史上传文件仍可访问。
- 临时目录可自动创建。
- 清理 session 后重新登录正常。
- 清理 cache 后页面正常。
- Git 状态不出现 session、error、upload 运行数据噪声。

部署验收：

- 新机器只需配置 `runtime.json`。
- 执行启动脚本可自动创建目录。
- 端口冲突能被启动脚本提示。
- 数据目录权限错误能被启动脚本提示。

## 13. 推荐执行顺序

建议按以下顺序推进：

1. 固定当前可运行版本。
2. 新增 `runtime.json`。
3. 新增目录诊断脚本。
4. 新增启动执行层。
5. 先迁移 `<quality_app>` 数据库目录。
6. 再迁移 `service_center` 数据库目录。
7. 迁移质量检查照片上传目录。
8. 配置临时目录自动创建和清理。
9. 明确临时目录不纳入核心备份。
10. 清理不再需要的软连接。

## 14. 当前建议

结合当前平台状态，不建议马上一次性取消全部软连接。

推荐先做：

- `runtime.json`
- `scripts/prepare_runtime_dirs.py`
- `scripts/run_platform.py`
- `applications/service_center/modules/runtime_paths.py`

然后优先改：

- `service_center` 认证数据库目录
- `<quality_app>` 业务数据库目录
- `<quality_app>` 质量检查照片上传目录

按临时目录处理：

- `sessions`
- `errors`
- `cache`

这些目录只需要启动层自动创建、检查权限和按策略清理，不需要按核心数据迁移。

这样改造风险最低，也最符合“保护真实业务数据”的目标。
