# 默认 web2py app 改造作业指导

版本：v1.0  
日期：2026-06-14  
适用范围：将仓库中的默认 web2py 风格 app 改造成平台统一规范 app

## 1. 目的

本作业指导用于把现有的默认 web2py 应用改造成当前平台统一模式，目标包括：

- 接入 `service_center` 统一认证和统一会话。
- 明确 app 自身业务库和认证库的边界。
- 将数据代码分离配置集中到模型启动层。
- 统一 `models` 编号语义。
- 统一菜单、权限、上传、备份和验收方式。

适用于以下类型的 app：

- 仍使用默认 `db.py`、`menu.py` 脚手架的 app。
- 仍将业务表和认证表混在一个 SQLite 中的 app。
- 仍依赖 `request.folder/uploads`、`request.folder/databases` 但没有集中路径配置的 app。
- 仍维护独立 Auth 而希望接入平台统一登录的 app。

## 2. 改造目标结构

推荐目标模型结构：

```text
models/
  00_app.py
  01_db.py
  20_tables.py
  30_menu.py
  40_permissions.py
  README.md
```

语义：

- `00_app.py`：app 基本信息
- `01_db.py`：共享模块和数据库初始化
- `20_tables.py`：私有业务数据
- `30_menu.py`：菜单导航
- `40_permissions.py`：私有权限

## 3. 改造前检查

开始前先确认下面内容：

### 3.1 目录和文件

检查 app 当前目录：

```bash
find applications/<app> -maxdepth 2 -type f | sort
find applications/<app> -maxdepth 2 -type d | sort
```

重点识别：

- `models/db.py`、`models/menu.py`
- `controllers/default.py`
- `private/appconfig.ini`
- `databases/`
- `uploads/`
- `sessions/`
- `errors/`
- `cache/`

### 3.2 数据库现状

确认当前 app 是否：

- 使用单独业务库
- 混用认证库和业务库
- 仍保留旧 `storage.sqlite`
- 有哪些 `.table` 迁移元数据

示例检查：

```bash
find applications/<app>/databases -maxdepth 1 -type f | sort
```

### 3.3 上传现状

扫描上传字段和手写上传逻辑：

```bash
rg -n "uploadfolder|Field\\(.*'upload'|response\\.download\\(|request\\.folder, 'uploads'|request\\.folder, \"uploads\"" applications/<app> -g '*.py'
```

### 3.4 权限现状

扫描：

- 是否有独立 `Auth`
- 是否已有自定义角色
- 是否已有 app 级权限装饰器

## 4. 改造步骤

### 步骤 1：备份数据

改造前必须备份：

- `applications/<app>/databases/`
- `applications/<app>/uploads/`
- 如果要接入统一认证，还要确认旧 `auth_user` 是否需要迁移

建议：

```bash
tar -czf <app>-before-refactor-$(date +%Y%m%d-%H%M%S).tar.gz applications/<app>/databases applications/<app>/uploads
```

### 步骤 2：梳理模型文件

把原始脚手架模型按平台规范拆分或重命名。

常见映射：

```text
db.py        -> 01_db.py
menu.py      -> 30_menu.py
其它业务表文件 -> 20_tables.py
权限相关逻辑   -> 40_permissions.py
新增         -> 00_app.py
```

如果原 app 文件很多，先不必大规模重构业务逻辑，只需要把启动顺序稳定下来。

### 步骤 3：创建 00_app.py

只放 app 基本信息，例如：

```python
# -*- coding: utf-8 -*-

APP_NAME = "示例应用"
APP_SUBTITLE = "业务副标题"
APP_ASSET_VERSION = "20260614-01"

response.title = APP_NAME
response.subtitle = APP_SUBTITLE
response.meta.author = "Your Team"
response.meta.description = "示例应用描述"
response.meta.keywords = "web2py, app"
```

要求：

- 不创建数据库连接
- 不定义业务表
- 不做角色初始化

### 步骤 4：改造 01_db.py

`01_db.py` 是整个改造的核心入口。

需要完成四件事：

1. 接入 `service_center` 统一认证
2. 显式配置运行数据目录
3. 创建业务库连接
4. 暴露共享变量给后续模型和控制器使用

推荐配置来源优先级：

1. 环境变量
2. `private/appconfig.ini`
3. 本地默认值

建议真实配置和模板分离：

- 提交到 Git：`private/appconfig.ini.example`
- 不提交到 Git：`private/appconfig.ini`

原因：

- 数据库和上传路径属于启动前配置，不能只放在数据库里。
- 生产环境经常需要按机器、容器或挂载点调整。
- 模型层只保留读取逻辑，真实路径由部署配置决定。

推荐结构：

```python
import os
import sys
from gluon.contrib.appconfig import AppConfig
from gluon.tools import Service, PluginManager

configuration = AppConfig(reload=True)

def _runtime_value(key, env_name, default):
    return os.environ.get(env_name) or configuration.get(key) or default

def _runtime_dir(key, env_name, root, dirname):
    return os.environ.get(env_name) or configuration.get(key) or os.path.join(root, dirname)

APP_RUNTIME_ROOT = _runtime_value(
    "runtime.root",
    "<APP>_RUNTIME_ROOT",
    request.folder,
)

APP_DB_DIR = _runtime_dir("runtime.databases_dir", "<APP>_DATABASES_DIR", APP_RUNTIME_ROOT, "databases")
APP_UPLOAD_DIR = _runtime_dir("runtime.uploads_dir", "<APP>_UPLOADS_DIR", APP_RUNTIME_ROOT, "uploads")
APP_SESSION_DIR = _runtime_dir("runtime.sessions_dir", "<APP>_SESSIONS_DIR", APP_RUNTIME_ROOT, "sessions")
APP_ERROR_DIR = _runtime_dir("runtime.errors_dir", "<APP>_ERRORS_DIR", APP_RUNTIME_ROOT, "errors")
APP_CACHE_DIR = _runtime_dir("runtime.cache_dir", "<APP>_CACHE_DIR", APP_RUNTIME_ROOT, "cache")

for path in (APP_DB_DIR, APP_UPLOAD_DIR, APP_SESSION_DIR, APP_ERROR_DIR, APP_CACHE_DIR):
    if not os.path.exists(path):
        os.makedirs(path)

shared_modules = os.path.join(request.folder, "..", "service_center", "modules")
if shared_modules not in sys.path:
    sys.path.append(shared_modules)

from service_center_auth import apply_service_center_auth
apply_service_center_auth(globals())

db_app = DAL(
    "sqlite://db_<app>.sqlite",
    migrate=True,
    pool_size=10,
    folder=APP_DB_DIR,
)

_db = db_app
service = Service()
plugins = PluginManager()
```

注意：

- `<APP>_RUNTIME_ROOT` 需要替换成实际 app 的大写环境变量名，例如 `<QUALITY_APP>_RUNTIME_ROOT`、`GJHD_RUNTIME_ROOT`。
- 生产环境推荐逐项显式指定 `<APP>_DATABASES_DIR`、`<APP>_UPLOADS_DIR` 等目录，系统迁移时更可控。
- 如果需要显式指定统一认证库位置，再增加 `<APP>_AUTH_ROOT`、`<APP>_AUTH_DATABASES_DIR`、`runtime.auth_root` 和 `runtime.auth_databases_dir`。
- 不要把路径散落到 `20_tables.py`、控制器、工具脚本里。
- 不要继续把业务表定义在认证库 `db` 上。
- 如果 app 还需要访问认证用户，继续使用 `db.auth_user`。
- `APP_DB_DIR`、`APP_UPLOAD_DIR` 可以直接被模型和控制器消费；`APP_SESSION_DIR`、`APP_ERROR_DIR`、`APP_CACHE_DIR` 需要结合 web2py 启动层、`masterapp` 或部署配置一起处理。

推荐 `private/appconfig.ini.example` 至少包含：

```ini
[db]
uri = sqlite://storage.sqlite
migrate = true
pool_size = 10

[db_<app>]
uri = sqlite://db_<app>.sqlite
migrate = true
pool_size = 10

[runtime]
root = /data/<web2py_service>/<app>
databases_dir = /data/<web2py_service>/<app>/databases
uploads_dir = /data/<web2py_service>/<app>/uploads
sessions_dir = /data/<web2py_service>/<app>/sessions
errors_dir = /data/<web2py_service>/<app>/errors
cache_dir = /data/<web2py_service>/<app>/cache
auth_root = /data/<web2py_service>/service_center
auth_databases_dir = /data/<web2py_service>/service_center/databases

[session]
masterapp = service_center
```

### 步骤 5：改造 20_tables.py

要求：

- 所有业务表都定义在 `_db` 或 `_app_db` 上
- 不再直接使用 `db.define_table()` 定义业务表
- 上传目录统一使用 `APP_UPLOAD_DIR`

示例：

```python
_app_db = _db

_app_db.define_table(
    "business_record",
    Field("name", "string", label="名称"),
    Field("attachment", "upload", uploadfolder=APP_UPLOAD_DIR),
)
```

跨库引用规则：

- 认证用户：`IS_IN_DB(db, "auth_user.id", "%(first_name)s")`
- 业务表：`IS_IN_DB(_app_db, "business_record.id", "%(name)s")`

### 步骤 6：改造 30_menu.py

如果 app 要接入平台统一顶部菜单，推荐像 `<quality_app>` 一样从 `service_center` 统一加载：

```python
import os

try:
    service_center_menu = os.path.join(request.folder, '..', 'service_center', 'models', 'xx_menu.py')
    exec(compile(open(service_center_menu, 'r', encoding='utf-8').read(), 'xx_menu.py', 'exec'), globals())
except Exception:
    response.menu = [
        (T('应用首页'), request.function == "index", URL("default", "index"), []),
    ]
```

如果需要 app 自己的入口，还要在：

```text
applications/service_center/modules/platform_nav.py
```

增加对应菜单记录。

### 步骤 7：改造 40_permissions.py

建议角色命名统一使用：

```text
<app>_super_admin
<app>_admin
<app>_全部_查看
<app>_全部_录入
<app>_全部_审核
<app>_<部门>_<操作>
```

权限文件中：

- 定义权限检查函数
- 定义权限装饰器
- 角色初始化保持幂等
- 不要在这里定义业务表

### 步骤 8：改造控制器

控制器需要统一改造以下内容：

1. 业务查询改成 `_db.<table>`
2. 上传逻辑统一使用 `_db` 对应 upload 字段
3. 下载逻辑改成：

```python
return response.download(request, _db)
```

4. 用户登录入口改成跳转统一认证：

```python
redirect(URL('service_center', 'default', 'user', args=request.args))
```

### 步骤 9：配置 appconfig.ini

推荐增加：

```ini
[session]
masterapp = service_center

[runtime]
root = /data/<web2py_service>/<app>
auth_root = /data/<web2py_service>/service_center
```

注意：

- 真实 `appconfig.ini` 不进 Git
- 应补充 `appconfig.ini.example`
- 如果业务库不是默认 `sqlite://db_<app>.sqlite`，再补 `[db_<app>]`
- 生产环境优先通过环境变量覆盖，例如：

```bash
export <APP>_RUNTIME_ROOT=/data/<web2py_service>/<app>
export <APP>_AUTH_ROOT=/data/<web2py_service>/service_center
export <APP>_DB_URI=sqlite://db_<app>.sqlite
```

### 步骤 10：补充文档

至少补下面两个文档：

1. `models/README.md`
2. app 级整合说明或运行说明

`models/README.md` 至少写清：

- 当前模型文件
- 编号语义
- 执行顺序
- 后续维护约定

### 步骤 11：增加只读运行路径诊断页

建议在 app 的设置页或诊断页增加只读运行状态区，用于展示：

- 当前业务数据库目录
- 当前上传目录
- 当前统一认证数据库目录
- 配置来源：环境变量、`private/appconfig.ini` 或默认值
- 目录是否存在、是否可写
- 是否为软链接，以及真实路径。软链接只作为旧部署兼容诊断，不作为新方案首选。
- 文件数量、目录大小
- 是否需要配合 web2py 启动层处理

推荐做法：

- 在 `01_db.py` 记录配置来源元信息，例如 `APP_CONFIG_SOURCES`
- 在控制器中整理 `runtime_status`
- 在页面中只读展示，不直接在线改 DAL 连接

不推荐：

- 通过前端页面即时改数据库目录并在当前请求中热切换 DAL
- 在多个控制器中各自重复探测目录状态

## 5. 推荐检查项

### 5.1 代码检查

```bash
python3 -m py_compile applications/<app>/models/*.py applications/<app>/controllers/*.py
```

### 5.2 路径检查

```bash
test -w <runtime-root>/databases
test -w <runtime-root>/uploads
test -w <auth-root>/databases
```

### 5.3 旧软链接兼容检查

```bash
find applications -maxdepth 2 -type l -print | while read p; do
  [ -e "$p" ] || echo "broken: $p -> $(readlink "$p")"
done
```

新迁移优先使用配置清单，不再主动创建软链接；这个检查只用于发现旧部署残留风险。

### 5.4 业务表检查

确认：

- 业务表在 app 业务库中
- 认证表在 `service_center` 认证库中

### 5.5 上传检查

确认：

- 上传文件落到 `APP_UPLOAD_DIR`
- 历史文件仍可通过 `response.download(request, _db)` 下载

### 5.6 页面诊断检查

确认：

- 设置页或诊断页能显示当前生效路径
- 能区分配置来源是环境变量、`appconfig.ini` 还是默认值
- 能识别目录缺失、不可写和软链接断链

## 6. 验收清单

基础验收：

- 未登录访问 app 自动跳转统一登录页
- 登录后可正常进入 app
- 平台顶部菜单可看到 app 入口

数据库验收：

- 认证数据写入当前认证运行目录的 `databases/storage.sqlite`
- 业务数据写入 app 自己的业务库
- 业务表不再混入认证库

上传验收：

- 新上传文件进入 app 运行数据目录
- 下载、预览、历史文件访问正常

运维验收：

- `databases/uploads/sessions/errors/cache` 不进入 Git
- 业务数据库和上传路径切换只需要修改 `01_db.py` 或 runtime 配置
- `sessions/errors/cache` 的最终落点要与 web2py 启动层配置保持一致
- 页面可只读查看当前生效路径和目录健康状态
- 数据备份和代码备份边界清晰

## 7. 典型问题

### 问题 1：认证库报 `sql.log` 找不到

原因：

- `service_center/databases` 是断链
- 数据已迁移但软链接未更新
- 或已经迁到外部目录，但模型/认证模块仍在读旧默认路径

处理：

- 新方案优先配置 `runtime.auth_databases_dir`
- 同时确认 `runtime.auth_root` 指向正确的 `service_center` 数据根目录
- 旧部署如果还有软链接，再单独检查是否断链

### 问题 2：业务表误写入认证库

原因：

- 仍使用 `db.define_table()` 定义业务表

处理：

- 改为 `_db.define_table()` 或 `_app_db.define_table()`

### 问题 3：上传文件落回 app 代码目录

原因：

- `uploadfolder` 仍写死 `request.folder/uploads`

处理：

- 改为统一引用 `APP_UPLOAD_DIR`

## 8. 推荐推进顺序

建议按以下顺序逐个 app 改造：

1. `<legacy_app>`
2. `<legacy_app>`
3. `<legacy_app>`
4. `<legacy_app>`
5. `<legacy_app>`
6. `<legacy_app>`
7. `<legacy_app>`
8. `<legacy_app>`
9. `<legacy_app>`

优先原则：

- 先改依赖平台认证但业务结构简单的 app
- 再改有上传、附件、权限的 app
- 最后改历史包袱较重的 app

## 9. 一句话原则

默认 web2py app 改造的核心原则是：把路径、认证、数据库连接集中到 `01_db.py`，把业务表留在 `20_tables.py`，把菜单和权限后置，最终让 app 只通过少数共享变量访问运行数据目录。
