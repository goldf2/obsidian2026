# web2py 服务通用文档

本仓库用于集中存放多个 web2py 项目复用的通用规范、运维流程、数据治理方案、部署治理路线图、项目级文档索引和历史参考资料。

历史来源目录:

- `/opt/<web2py_service>/applications/docs`
- `/opt/<web2py_service>/applications/<quality_app>/docs`
- `/opt/<web2py_service>/applications/service_center/docs`

## 目录说明

| 目录 | 用途 | 使用规则 |
| --- | --- | --- |
| 根目录 | 平台开发规范、菜单架构规范、数据代码分离规范、PaaS 部署治理路线图、备份规范、CI/CD 流程 | 当前高频使用的通用规范 |
| `创新系统架构/` | 系统架构图、导航和页面总览 | 作为新项目理解平台能力和页面结构的入口 |
| `故障排查/` | 常见错误排查指南 | 作为跨 app 问题处理参考 |
| `80-模板/` | 当前仍在使用的项目文档模板 | 作为项目文档初始化模板 |
| `projects/` | 每个实际 web2py 项目的项目级文档 | 记录项目事实、应用清单、运行数据和规范对照 |
| `90-历史参考/` | 旧方案、旧目录结构、旧模板和 service_center 历史资料 | 只作追溯参考，不直接作为当前执行依据 |

## 维护规则

- 通用规范优先维护在根目录或明确分类中，具体项目只保留业务专属文档、项目差异说明或指向通用规范的索引。
- 涉及生产路径时，文档应同时说明“通用规则”和“项目实例”，避免把某一个项目路径误当成所有项目路径。
- 菜单架构以 `菜单架构统一规范.md` 为当前执行口径，导航总览只登记入口、页面和权限清单。
- 数据代码分离以 `数据代码分离统一规范.md` 为当前执行口径，不再单独维护生产数据分离示例文档。
- 每个实际 web2py 服务项目必须在 `projects/` 下有同名文件夹，并维护项目级文档。
- `90-历史参考/` 下的文档可能包含旧路径、旧部署方式或旧目录结构，不作为当前执行依据；需要 PaaS、容器化或传统部署治理时，以 `数据存储与PaaS部署治理路线图.md` 和根目录 CI/CD、备份规范为准。
- 新增共享文档应先判断是否属于当前高频文档；旧方案、旧模板和低频追溯资料放入 `90-历史参考/`。
